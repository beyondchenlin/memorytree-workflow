# 方案 2 说明：Caddy + GitHub Pages

## 文档目的

本文档用于团队讨论 MemoryTree 报告站点的推荐部署方案。

目标：

- 本机可以长期稳定访问报告站点
- 外部设备或团队成员可以通过固定网址访问
- 不依赖长期打开终端窗口
- 尽量复用当前 MemoryTree 已有能力，不把简单预览工具硬扛成正式托管方案

说明：

- 本文档是方案说明，不代表当前仓库已经全部按此方案落地
- 本文档只讨论部署与访问方式，不讨论业务代码修改

## 方案结论

推荐方案：

- 本机常驻访问：使用 `Caddy`
- 对外发布：使用 `GitHub Pages`
- 内容生产：使用 `MemoryTree heartbeat` 和 `report build`

一句话概括：

`MemoryTree` 负责生成静态报告，`Caddy` 负责本机长期可访问，`GitHub Pages` 负责公网长期可访问。

## 为什么选这个方案

当前仓库里的 `memorytree report serve` 更适合“本地临时预览”，不适合直接承担“本机常驻 + 公网发布”两个目标。

原因：

- `report serve` 是一个临时启动的本地 HTTP 进程
- 终端关闭后服务就会停止
- 它本质上是轻量预览工具，不是正式的网站托管方案
- heartbeat 的职责是定时处理 transcript 和报告生成，不是长期监听端口提供网页

因此，更合理的拆分是：

- `MemoryTree` 负责“生成内容”
- `Caddy` 负责“本机常驻托管”
- `GitHub Pages` 负责“公网托管”

## 方案结构

### 1. 内容层：MemoryTree

MemoryTree 的职责：

- 导入 transcript
- 清洗与归档 transcript
- 在需要时生成 `Memory/07_reports`
- 在需要时将生成后的静态站点发布到专门分支

MemoryTree 生成的是静态网站文件，核心目录通常是：

```text
Memory/07_reports/
```

这里面是 HTML、CSS、JS、RSS 等静态文件。

### 2. 本机访问层：Caddy

Caddy 的职责：

- 长期监控并托管 `Memory/07_reports`
- 在本机暴露一个固定地址，例如：

```text
http://localhost:10010/
```

- 作为系统服务长期运行，不依赖终端窗口

可以把 Caddy 理解成“本机网站服务员”，专门负责把某个文件夹变成一个长期可访问的网站。

### 3. 对外发布层：GitHub Pages

GitHub Pages 的职责：

- 将生成好的静态站点发布为公网网址
- 提供稳定的外部访问入口
- 让团队成员、其他设备、手机和平板可以直接访问

可以把 GitHub Pages 理解成“对外托管平台”，专门负责公网访问。

## 典型工作流

日常运行时的流程如下：

1. 开发者继续在仓库中工作。
2. heartbeat 按周期运行，处理 transcript。
3. 如果配置允许并且有新内容，MemoryTree 生成或更新 `Memory/07_reports`。
4. Caddy 持续托管 `Memory/07_reports`，因此本机访问 `localhost:10010` 时能直接看到站点。
5. 如果配置了发布分支，更新后的静态站点会被推送到发布分支。
6. GitHub Pages 将该分支内容发布为固定公网地址。

结果是：

- 本机有本地入口
- 公网有对外入口
- 两边共用同一份静态产物

## 本机访问与公网访问的区别

### 本机访问

典型地址：

```text
http://localhost:10010/
```

特点：

- 访问速度快
- 不依赖公网
- 适合自己日常查看
- 由 Caddy 常驻提供服务

### 公网访问

典型地址：

```text
https://<your-site>.github.io/<repo-or-site-path>/
```

特点：

- 任何联网设备都可访问
- 适合团队分享
- 不依赖你的电脑终端是否打开
- 由 GitHub Pages 托管

## 这个方案相对其他方案的优势

### 相对 “只用 report serve”

优势：

- 不需要长期打开终端
- 本机访问更稳定
- 更接近正式服务，而不是临时预览
- 更适合团队长期使用

### 相对 “IIS + GitHub Pages”

优势：

- `Caddy` 是跨平台的
- Windows、macOS、Ubuntu 都可以使用
- 如果未来更换主机环境，迁移成本更低
- 配置和维护通常比 IIS 更轻

### 相对 “本机服务直接暴露到公网”

优势：

- 风险更低
- 不需要自己处理公网暴露、端口转发、家庭宽带限制等问题
- GitHub Pages 更适合静态网站发布
- 不把个人电脑直接作为公网生产服务入口

## 这个方案的代价与限制

需要接受以下事实：

- 本机服务和公网服务是两套出口，不是同一个进程
- 本机常驻依赖 Caddy，不是依赖 `report serve`
- 公网发布依赖 GitHub Pages，不是依赖本机机器长期在线
- 如果 heartbeat 没有生成新报告，Caddy 和 GitHub Pages 展示的仍然是最近一次生成后的内容

这不是缺点，而是职责拆分后的自然结果。

## 为什么不直接把 report serve 当成常驻方案

主要原因：

- `report serve` 是一个临时 Node 进程
- 它适合本地开发预览
- 它不是专门为长期服务托管设计的
- 直接把它拿来承担公网服务，运维成本和风险都会上升

因此，更合理的使用方式是：

- `report serve` 用于临时检查
- `Caddy` 用于本机常驻
- `GitHub Pages` 用于对外发布

## 安全与信息边界

这是团队必须重点讨论的部分。

如果站点对外发布，需要明确：

- 哪些报告内容可以公开
- transcript 中是否可能含有敏感信息
- 是否需要区分“本机版”和“对外版”
- 团队是否接受报告默认可被外部访问

建议：

- 如果内容存在敏感风险，应先明确公开范围再推进公网发布
- 不应默认把所有本地可见内容直接同步到公网

## 适用场景

这个方案最适合以下场景：

- 开发者希望自己在本机随时查看报告
- 团队希望通过固定网址共享报告
- 希望避免“每次开网页前都要手动起服务”
- 希望后续有跨平台迁移能力

## 不适合的场景

以下情况不建议优先采用本方案：

- 报告内容高度敏感，不能接受公网暴露风险
- 团队只需要偶尔本地预览，不需要常驻或分享
- 项目明确只运行在 Windows 且长期不会迁移，团队更偏向深度使用 IIS

## 推荐的团队认知模型

团队讨论时，建议统一用下面这套理解：

- `MemoryTree` 不是网站服务器，它是“内容生产系统”
- `Caddy` 不是内容生成器，它是“本机网站托管器”
- `GitHub Pages` 不是本机服务，它是“公网静态发布平台”

三者职责不同，但可以自然拼接成一条完整链路。

## 最终建议

如果团队同时需要：

- 本机常驻访问
- 对外发布访问
- 尽量少依赖手工终端操作
- 未来保留跨平台迁移能力

那么 `Caddy + GitHub Pages` 是当前最均衡、最稳妥的方案。

它不是最“省组件”的方案，但通常是长期维护成本最低、边界最清晰的方案。
