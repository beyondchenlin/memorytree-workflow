# 记忆分支单一真源两阶段改造方案

## 1. 背景

当前 MemoryTree 的实现采用了「开发目录 + memory worktree + 全局归档」三层结构。

它已经具备 worktree 隔离和 `memorytree` 分支自动提交能力，但产品语义仍然保留了两套承载方式：

- `memorytree` 分支承担一部分共享记忆
- 开发目录中的 `AGENTS.md` 和 `Memory/**` 仍被视为可见且可提交的仓库内容

这会带来三个直接问题：

- 共享记忆的唯一来源不清晰，容易形成双份承载
- 开发目录容易出现敏感内容误提交
- `5m heartbeat + 30m refresh` 的模型增加理解和维护成本

## 2. 新目标

本次改造要把产品模型统一为：

- `memorytree` 分支和对应的 memory worktree 是唯一共享记忆源
- 开发目录中的 `AGENTS.md` 和 `Memory/**` 全部都是本地缓存镜像
- 开发目录缓存允许被 heartbeat 持续同步和覆盖，但默认不参与常规 Git 提交
- heartbeat 的外部语义统一为固定短周期执行，不再对外强调单独的 `30m refresh` 模型
- 所有 MemoryTree 内容最终都应完整进入 `memorytree` 分支，而不是只进入其中一部分

## 3. 为什么要改

### 3.1 降低误提交和敏感信息暴露风险

当前主目录中的 `Memory/**` 很容易出现在 `git status` 中。只要用户习惯性提交，就可能把 transcript、报告和其他记忆内容一起推到正常开发分支。

把开发目录改成缓存层之后，常规开发提交流程就不再默认携带这些内容。

### 3.2 建立单一真源

只要主目录和 `memorytree` 分支都在承担「正式记忆承载」职责，系统就会长期处在双份状态，难以判断哪一份才是应该被共享、备份和恢复的权威版本。

将共享职责收敛到 `memorytree` 分支后，语义会明显更稳定。

### 3.3 简化产品心智模型

现有的 `5m heartbeat + 30m refresh` 设计会让用户反复追问：

- 哪个动作在 `5m` 做
- 哪个动作在 `30m` 做
- 为什么主目录看起来仍然像 `5m` 在更新

如果主目录只是缓存镜像，那么产品可以统一解释为：

- heartbeat 固定周期运行
- 产出写入 `memorytree` 分支
- 开发目录只是同步后的本地镜像

### 3.4 让同事更容易理解职责边界

对于团队协作来说，更清晰的边界是：

- 要共享和传播，去看 `memorytree` 分支
- 要本地查看和继续工作，直接看当前开发目录的缓存镜像
- 不再要求普通开发分支天然承载所有记忆文件

## 4. 范围边界

本方案默认采用下面的边界：

- `AGENTS.md`、`Memory/01-07` 都属于 MemoryTree 内容
- 这些内容都应进入 `memorytree` 分支
- 开发目录中的对应文件只作为缓存镜像存在
- 不要求重写旧 Git 历史
- 不要求一开始就做复杂迁移兼容

## 5. 第一阶段：必须完成

### 1. 冻结新的产品规则

目的：

- 统一团队口径，避免后续实现过程中再次摇摆

本项要做的事：

- 明确写死 `memorytree` 分支是唯一共享记忆源
- 明确写死开发目录中的 `AGENTS.md` 和 `Memory/**` 只是缓存镜像
- 明确写死主开发分支不再承担记忆传播职责

完成标准：

- README、设计文档、规则文档不再出现双份承载的模糊表述

### 2. 去掉 `30m refresh` 的对外产品语义

目的：

- 降低模型复杂度
- 消除当前实现与说明之间的理解偏差

本项要做的事：

- 不再把 `refresh_interval = 30m` 作为推荐默认行为对外描述
- Quick Start、CLI help、README 示例不再强调 `5m heartbeat + 30m refresh`
- 把主目录缓存同步解释为 heartbeat 工作流的一部分，而不是独立产品能力

完成标准：

- 用户不再需要理解一个额外的 `30m refresh` 概念

### 3. 补齐 `memorytree` 分支的提交闭环

目的：

- 让 `memorytree` 分支真正承载全部共享记忆，而不是只承载其中一部分

本项要做的事：

- 让 `AGENTS.md` 进入自动提交范围
- 让 `Memory/01-05` 的同步结果在需要时进入自动提交范围
- 重新确认 `Memory/06` 的 clean、manifest、raw、full 等子路径如何统一进入 `memorytree` 分支
- 不再让「某些文件同步过去了，但没有被自动 commit」成为常态

完成标准：

- memory worktree 中形成的完整 MemoryTree 内容都能稳定进入 `memorytree` 分支

### 4. 调整 heartbeat 的执行顺序

目的：

- 避免 `memorytree` 分支只保存半成品状态

本项要做的事：

- 先同步开发目录缓存到 worktree
- 再导入 transcript
- 再生成 report
- 最后统一 commit/push

完成标准：

- 一轮 heartbeat 结束后，远程 `memorytree` 分支就是该轮完整产出的快照

### 5. 对开发目录执行一次性退跟踪切换

目的：

- 让开发目录中的缓存镜像从下一版本开始默认不参与正常 Git 提交流程

本项要做的事：

- 在 `.gitignore` 中加入 `AGENTS.md` 和 `Memory/**` 的规则
- 将当前已跟踪的 `AGENTS.md` 和 `Memory/**` 从 Git index 中移除
- 保留本地文件，不删除开发目录缓存内容
- 验证 heartbeat 仍然能在开发目录写入和同步这些缓存文件

完成标准：

- 开发目录中的缓存镜像继续存在，但默认不会进入常规开发提交

### 6. 统一 README、设计文档和规则文档

目的：

- 防止实现改了、说明还停留在旧模型

本项要做的事：

- 更新 README 的产品定位、架构说明、Heartbeat 说明、配置说明
- 更新 worktree 设计文档中的同步语义和缓存语义
- 更新 Git policy、update rules、transcript archive 等规则文档
- 清除「开发目录里的 Memory 文件默认可提交」这类旧表述

完成标准：

- 同事只看文档就能理解新的单一真源模型

## 6. 第二阶段：后续收口

### 7. 清理 `refresh_interval` 的配置和 CLI 残留

目的：

- 把第一阶段的产品语义调整真正落实到配置模型和命令界面

本项要做的事：

- 评估是否彻底移除 `refresh_interval`
- 如果短期保留兼容字段，也要去掉它的主导地位
- 清理 CLI 参数、帮助文本、配置示例中的残留说明

完成标准：

- 对外界面不再把 `refresh_interval` 当成核心能力

### 8. 重写 `init` / `upgrade` / onboarding 文案

目的：

- 让新用户从第一次接触起就理解「memorytree 分支是真源，开发目录是缓存」

本项要做的事：

- 调整初始化与升级提示
- 调整 Quick Start 的解释方式
- 调整技能文档和中文说明文档

完成标准：

- 新用户不会再把主目录误认为正式共享记忆层

### 9. 做跨平台与 E2E 回归验证

目的：

- 保证新模型在 Windows、macOS、Linux 上都能稳定工作

本项要做的事：

- 验证 worktree 同步、退跟踪后的主目录状态、heartbeat 提交闭环
- 验证 report、recall、discover、daemon quick-start 等关键路径
- 覆盖 main directory 缓存镜像持续写入的场景

完成标准：

- 新的单一真源模型有跨平台自动化验证支撑

### 10. 评估是否处理旧 Git 历史中的敏感内容

目的：

- 明确当前版本切换是否只面向未来，还是还要额外清理历史

本项要做的事：

- 判断是否需要对历史中的 `AGENTS.md` / `Memory/**` 做额外风险评估
- 判断是否需要清洗旧历史或只从下一版本开始执行新规则

完成标准：

- 团队对历史风险边界有明确结论，不再含糊

#### 当前评估结论（2026-03-24）

- 基于仓库历史扫描，`main` 历史中已有 `10` 个提交触达过 `AGENTS.md` / `Memory/**`
- 其中 `main` 历史里已有 `2` 个提交触达过 `Memory/06_transcripts/raw/**`
- 按全部可达引用扫描，`AGENTS.md` / `Memory/**` 已出现在 `66` 个提交中，说明这不是只存在于单个临时分支的偶发情况
- 对 `AGENTS.md`、`Memory/06_transcripts/raw/**`、`Memory/06_transcripts/full/**` 做历史 grep 时，确实能打出较多“敏感模式”命中
- 但这些命中目前仍是启发式结果，里面混有文档示例、评审文本、旧方案记录等内容，不能直接等同于“已确认存在仍然有效的真实密钥”

#### 结论

- 当前版本切换按“面向未来”执行：不把重写旧 Git 历史作为本阶段默认动作
- 已完成的退跟踪、原始 transcript 上传许可、单一真源分支隔离，负责降低后续新增暴露面；它们不会自动抹除旧历史
- 如果后续确认旧历史中存在真实且仍有效的凭据、受监管个人数据、或必须清理的敏感 transcript，再走单独的历史清洗/应急流程

#### 触发历史清洗的条件

- 已确认旧历史中包含仍然有效的 API key、token、密码、私钥或其他可直接利用的凭据
- 已确认旧历史中包含必须删除的受监管数据或高敏个人信息
- 仓库已经公开传播，或已有合规 / 安全审计要求明确要求清洗历史

#### 历史清洗的建议顺序

1. 先撤销或轮换已暴露凭据，再讨论 Git 历史处理
2. 暂停继续扩散相关 mirror / transcript 推送，避免新提交继续放大暴露面
3. 评估影响范围：`main`、开放分支、tag、release 产物、Actions artifact、fork
4. 仅在仓库所有者明确批准后，再执行 `git filter-repo` / BFG 一类的历史重写
5. 历史重写后同步通知协作者重新 clone 或强制对齐，并复核 `.gitignore`、de-track、raw 上传策略是否仍生效

## 7. 实施顺序建议

建议按下面顺序推进：

1. 先冻结产品规则
2. 再补齐 `memorytree` 分支提交闭环
3. 再调整 heartbeat 执行顺序
4. 再做开发目录退跟踪切换
5. 最后统一文档和规则文件
6. 第二阶段再清理配置残留、补 onboarding 和做回归验证

原因：

- 如果先退跟踪、后补闭环，中间阶段会出现 `memorytree` 分支尚未完整接住全部记忆的问题
- 先把真源逻辑补齐，再切换开发目录为缓存层，风险更低

## 8. 非目标

当前方案不把下面这些事作为第一阶段前提：

- 不重写旧 Git 历史
- 不做复杂的多版本迁移兼容工程
- 不依赖定时 branch merge
- 不让开发目录重新承担共享记忆职责

## 9. 同步给同事时的简版说明

可以用下面这段话向同事解释：

MemoryTree 接下来要从「开发目录和 memorytree 分支共同承载记忆」切换成「memorytree 分支单一真源 + 开发目录本地缓存镜像」。这样做的目的，是减少主开发分支误提交敏感信息的风险，统一共享记忆来源，并简化 heartbeat 的产品模型。第一阶段先完成真源闭环、退跟踪切换和文档统一，第二阶段再清理配置残留与补足回归验证。

## 10. 可执行修订方案 V2（按当前代码基线）

本节是对上文实施顺序的落地化修订，目标是避免出现「开发目录已退跟踪，但 memorytree 分支尚未完整接住内容」的中间风险窗口。

### 10.1 开工门槛（Gate）

在执行开发目录退跟踪之前，必须同时满足：

- G1：自动提交范围已覆盖 `AGENTS.md` 与 `Memory/01-07`（按策略允许的子路径）
- G2：即使「本轮无新 transcript」，仍可提交本轮 context/report 变更
- G3：heartbeat 单轮顺序已调整为 `sync context -> import -> build report -> commit/push`
- G4：新增单测与 E2E 用例已通过（含 Windows/macOS/Linux）

只要任一 Gate 未满足，就不得进入退跟踪切换。

### 10.2 第一阶段（必须完成）改为 3 个子阶段

#### 阶段 1A：规则冻结与文档对齐（先做）

目标：

- 锁定「memorytree 分支单一真源」口径，避免实现过程中反复回摆

动作：

- 更新 README、设计说明、规则文档中的目标语义
- 明确开发目录是缓存镜像，不承担共享传播职责

完成标准：

- 新文档口径与目标模型一致，不再出现双承载描述

#### 阶段 1B：提交闭环与执行顺序修补（必须一次打通）

目标：

- 让 memory worktree 中形成的 MemoryTree 内容在同一轮 heartbeat 内形成完整快照

动作：

- 扩大自动提交扫描范围：不再只看 `Memory/`，要覆盖托管路径（含 `AGENTS.md`）
- 提交时使用可覆盖 ignore 的 staging 方案，确保缓存退跟踪后依然可由 memory 分支接住新增文件
- 移除「`importedCount === 0` 直接 return」导致的漏提交路径
- 把 report 生成前置到 commit/push 之前
- 统一 `Memory/06` 子路径（clean / manifests / raw / full）的提交策略与许可策略

完成标准：

- 一轮 heartbeat 结束后，`memorytree` 分支可直接代表该轮完整快照
- 不再出现「已同步但未 commit」的常态

#### 阶段 1C：开发目录退跟踪切换（仅在 Gate 全绿后执行）

目标：

- 让开发目录缓存默认不进入常规开发提交流程

动作：

- 在开发分支 `.gitignore` 增加 `AGENTS.md` 与 `Memory/**` 规则
- 对开发分支执行一次性 de-track（从 index 移除，保留本地文件）
- 验证 heartbeat 仍能持续写入开发目录缓存镜像

完成标准：

- 开发目录缓存可读可用，但默认不参与常规开发提交
- `memorytree` 分支仍可稳定接住所有托管记忆内容

### 10.3 第二阶段（收口）

目标：

- 将第一阶段的语义调整落实到 CLI/配置/测试与 onboarding

动作：

- `refresh_interval` 降级为兼容字段：对外帮助文本、Quick Start、README 不再强调
- `init` / `upgrade` / onboarding 文案改为「分支真源 + 开发目录缓存」
- 清理历史文档中 Python-era 与旧模型描述
- 补齐跨平台 E2E（退跟踪后同步、提交闭环、report、recall、discover）

完成标准：

- 对外界面不再把 `refresh_interval` 作为核心能力
- 新用户首次接触即理解单一真源模型

## 11. 建议里程碑（便于排期）

- M1（语义冻结）：文档口径统一，不再双承载
- M2（闭环修补）：同轮完整快照达成，新增测试通过
- M3（退跟踪落地）：开发目录缓存化切换完成且可持续运行
- M4（外部收口）：CLI/README/onboarding 全量更新并完成回归

## 12. 回滚与止损策略

若阶段 1C 执行后出现提交缺口或数据漂移，立即：

- 暂停退跟踪继续扩散（冻结新仓库切换）
- 回退开发分支 `.gitignore` 的相关条目
- 恢复必要路径到受控跟踪状态，直到阶段 1B 闭环修补重新验收通过

回滚原则：

- 优先保证 `memorytree` 分支持续可写且可追溯
- 禁止在真源不稳定时继续扩大缓存化覆盖范围
