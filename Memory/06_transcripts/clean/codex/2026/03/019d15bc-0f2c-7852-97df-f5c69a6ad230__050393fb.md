你现在想要的，已经不是单纯“修一个小 bug”，而是在把 MemoryTree 的产品语义往“更隐私、更少打扰主开发目录”的方向收紧。这个方向是合理的，而且和你前面担心的“用户误提交敏感信息”是完全一致的。

但要分清两件事：

1. 当前文档里的原始设计，并不是你现在想要的这个模式。  
在 [docs/worktree-heartbeat-upgrade-plan.md:113](D:/demo1/memorytree-workflow/docs/worktree-heartbeat-upgrade-plan.md:113) 之后，原设计明确写的是：
- `Memory/06`、`Memory/07` 会回写到主目录
- 回写到主目录后，默认就是“用户可见、也可提交”的
- `30m refresh` 是一个比 `5m heartbeat` 更慢的主目录刷新节奏

2. 当前代码实现，又和这个原设计不完全一致。  
在 [heartbeat.ts:111](D:/demo1/memorytree-workflow/src/heartbeat/heartbeat.ts:111) 和 [heartbeat.ts:145](D:/demo1/memorytree-workflow/src/heartbeat/heartbeat.ts:145)，代码是：

- `refreshDue = 30m`

这意味着现在主目录回写基本会跟着 `5m heartbeat` 走，不会老老实实等 `30m`。所以如果我们按“旧设计”来判断，这里更像是实现和设计文档不一致，不是你配置写错了。

**你现在的新需求**
你刚刚给出的新版目标，我建议我们先冻结成下面这版：

1. 不再保留 `30m refresh` 这个概念，统一成 `5m`。  
2. heartbeat 仍然每 `5m` 跑，但“是否回写主目录”要看主目录当前分支。  
3. 如果主目录当前是 `main` 或长期开发分支，就允许把 memory 结果同步回主目录。  
4. 如果主目录当前不是这些分支，就只更新 memory worktree / `memorytree` 分支，不回写主目录。  
5. 主目录里的 `AGENTS.md` 和 `Memory/**` 希望默认不要参与普通开发提交，避免用户误推敏感内容。  
6. GitHub Pages 希望尽量自动，但可以接受某些首次配置仍需用户确认。

这个方案是自洽的，我认为方向上是比现在更清晰的。

**有一个关键技术点要提前说明**
你提到“把主目录的 `Memory` 放进 `.gitignore`，就不提交”。这个想法对“新出现的未跟踪文件”有效，但对“已经被 Git 跟踪过的文件”不够。

当前 [`.gitignore`](D:/demo1/memorytree-workflow/.gitignore) 只忽略了 `Memory/07_reports/`。如果以后把 `Memory/**` 和 `AGENTS.md` 加进去：

- 对新文件有帮助
- 但对已经进入 Git 历史、已经被跟踪的文件，不会自动失效
- 也就是说，单靠 `.gitignore` 不能完整解决“主目录同步但不提交”的问题

所以这件事如果以后真要落地，机制上通常得是：
- 要么从产品层面让这些路径默认不被主目录 Git 跟踪
- 要么用更强的本地防护机制，而不只是 `.gitignore`

这点很重要，因为它决定了我们后面设计时不能把 `.gitignore` 当成唯一方案。

**长期开发分支怎么判断**
这块我给你一个比较稳的建议。

对“通用产品”来说，最靠谱的顺序应该是：

1. 最优先：项目显式配置  
例如每个项目配置一个“允许回写主目录的分支列表”。这是最稳的。

2. 自动推断第一层：仓库默认分支  
也就是 `origin/HEAD` 指向的默认分支，比如 `main`。

3. 自动推断第二层：CI / workflow 里显式声明的长期分支  
你这个仓库现在就有很明显的信号：  
[ci.yml](D:/demo1/memorytree-workflow/.github/workflows/ci.yml:1) 和 [e2e.yml](D:/demo1/memorytree-workflow/.github/workflows/e2e.yml:1) 都把 `main` 和 `ts` 当成主要分支。

4. 最后的保守回退：如果推断不清楚，就只认默认分支  
宁可少同步，也不要误把 feature 分支当成长线分支。

所以对“这个仓库”本身，我现在会把长期开发分支先判断为：
- `main`
- `ts`

这个判断是有仓库内证据支撑的，不是拍脑袋。

**GitHub Pages 这块的判断**
现在代码已经支持“有配置就自动发 gh-pages 分支”，见 [build.ts:247](D:/demo1/memorytree-workflow/src/report/build.ts:247) 和 [github-pages.ts](D:/demo1/memorytree-workflow/src/report/deploy/github-pages.ts)。  
所以“自动发布内容”是可以的。

但“自动帮用户把 GitHub 仓库的 Pages 设置开好”这件事，当前实现还没有接 GitHub API / GitHub CLI 这层能力。  
所以它现在是：
- 自动推送 `gh-pages` 分支：可以
- 自动完成 GitHub 仓库后台 Pages 开关配置：还不是现成能力

**我对当前局面的结论**
1. 旧设计里，`30m refresh` 是有明确产品语义的。  
2. 当前实现把主目录回写基本做成了 `5m`，所以这部分更像实现偏离旧设计。  
3. 但你现在其实想换一套更合理的新设计：统一 `5m`，并且只在 `main/长期分支` 回写主目录。  
4. 这个新设计是成立的，而且比现状更符合“隐私优先”的目标。  
5. 真正难点不在 heartbeat，而在“主目录同步后如何默认不参与普通 Git 提交”，这里 `.gitignore` 不够，需要更完整的策略。
