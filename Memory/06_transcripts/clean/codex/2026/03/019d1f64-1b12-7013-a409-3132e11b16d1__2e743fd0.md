有，而且“源头方案”不是改 `git add`，而是改流水线设计，让每轮只产生真正增量。

最根本的做法是这 4 件事一起做：

1. **入口去重前移（最关键）**  
现在每轮都在反复“Imported 70/72 transcript(s)”，说明扫描阶段把历史文件反复当作本轮输入。  
源头解法：维护一个持久化游标/指纹表（`client + source_path + raw_sha256`），已处理且内容未变就直接跳过，不进入 import/report/commit。

2. **导入改成“仅变更写入”**  
对 `clean/full/manifest` 做内容比对（hash 或字节比对），相同就不重写文件，不更新时间戳。  
这样 Git 侧不会出现“看起来每轮都变了”的假变更。

3. **报告改成增量构建**  
不要每 5 分钟重生成整套 `Memory/07_reports`。  
只重建受影响会话页 + 必要索引页；或把报告完全移到单独产物通道（不参与 heartbeat 主提交）。

4. **失败回压（防滚雪球）**  
如果 commit/push 连续失败，heartbeat 不应继续导入和产出大量新文件。  
应进入“保护模式”：暂停写入、只告警，等故障解除再恢复。

一句话总结：  
**把系统从“全量重算+末端提交”改为“入口判重+全链路增量+失败回压”**，这才是从源头解决“越跑越堆、最终爆掉”的方案。
