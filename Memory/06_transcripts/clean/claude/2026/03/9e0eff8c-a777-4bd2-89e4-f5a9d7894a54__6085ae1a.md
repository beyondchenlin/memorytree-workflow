# Code Review: Python-to-TypeScript Migration

## 1. CRITICAL Issues

### 1.1 `normalizeTimestamp` -- Numeric threshold diverges from Python

**File:** `src/transcript/common.ts:43`
**Severity:** CRITICAL

The Python version passes any `int`/`float` directly to `datetime.fromtimestamp()`, which interprets all values as Unix seconds. The TypeScript version applies a heuristic:

```ts
const ms = value < 1e12 ? value * 1000 : value
```

This means a timestamp like `1700000000000` (milliseconds) is treated as-is, but the Python version would treat the same number as seconds (year ~55850). While the heuristic is reasonable, it is a **behavioral divergence** from the Python original. Any upstream system that stores timestamps as seconds with very large values (e.g., nanosecond-resolution timestamps divided down) will be handled differently. More critically, values at exactly `1e12` (September 2001 in ms, September 33658 in seconds) land on the wrong side.

### 1.2 `sha256File` reads entire file into memory

**File:** `src/transcript/common.ts:32-35`
**Severity:** CRITICAL

```ts
export function sha256File(filePath: string): string {
  const data = readFileSync(filePath)
  return createHash('sha256').update(data).digest('hex')
}
```

The Python version streams the file in 1 MB chunks. This TypeScript version loads the entire file into RAM. For large transcript files (some Codex sessions can be 50+ MB), this can cause OOM crashes. This should use `createReadStream` with a streaming hash.

### 1.3 `parseIsoTimestamp` -- timezone handling divergence

**File:** `src/transcript/common.ts:80-86`
**Severity:** CRITICAL

The Python version explicitly handles timezone-naive ISO strings by assigning `timezone.utc`:
```python
if parsed.tzinfo is None:
    return parsed.replace(tzinfo=timezone.utc)
return parsed.astimezone(timezone.utc)
```

The TypeScript version converts `Z` to `+00:00` but then passes the string to `new Date()`, which interprets timezone-naive strings (no offset, no `Z`) as **local time** per the JS spec. The Python version treats them as UTC. This means a timestamp like `"2024-06-15T10:30:00"` will be parsed as 10:30 local time in TS but 10:30 UTC in Python -- a behavioral divergence on all non-UTC systems.

### 1.4 `installCron` passes no stdin data to `crontab -`

**File:** `src/cli/cmd-daemon.ts:134-135`
**Severity:** CRITICAL

```ts
execCommand('crontab', ['-'], { allowFailure: false })
// Use stdin approach - write temp file and pipe
const tmpFile = resolve(homedir(), '.memorytree', '.crontab.tmp')
```

The first `execCommand('crontab', ['-'])` call will hang or install an empty crontab because `execFileSync` provides no stdin. Then it does the temp file approach. The first call should be removed entirely.

## 2. HIGH Issues

### 2.1 `db.ts` -- WAL mode is a no-op with in-memory sql.js

**File:** `src/transcript/db.ts:114`
**Severity:** HIGH

```ts
db.run('PRAGMA journal_mode=WAL')
```

sql.js operates entirely in memory (loaded from a Buffer, exported back to a Buffer). WAL mode is meaningless here since there is no actual file-level journal. More importantly, the read-export-write cycle is **not atomic**: if the process crashes between `db.export()` and `writeFileSync()`, data is lost. Also, concurrent writers will silently overwrite each other since there's no file-level locking. The Python version using native sqlite3 has proper file locking via the OS.

### 2.2 Missing `global_manifest_path` column in DB

**File:** `src/transcript/db.ts:44-52`
**Severity:** HIGH

The `DATA_COLUMNS` list does not include `global_manifest_path`, but `ManifestEntry` in `src/types/transcript.ts:43` has it. The Python DB schema also includes this column. Any `upsertSearchIndex` call silently drops the `global_manifest_path` field. When `recall.ts` queries `SELECT *`, the column will be missing from results.

### 2.3 `slugify` strips differently from Python

**File:** `src/transcript/common.ts:27-29`
**Severity:** HIGH

Python: `slug.strip("-._")` strips characters from both ends, removing **any combination** of `-`, `.`, `_` from each end.

TypeScript: `.replace(/^[-._]+|[-._]+$/g, '')` -- this is actually equivalent. **No issue here upon closer inspection.**

However, the ASCII filtering differs:
- Python: `value.encode("ascii", "ignore").decode("ascii")` -- drops non-ASCII bytes silently
- TypeScript: `value.replace(/[^\x00-\x7F]/g, '')` -- drops non-ASCII characters

These are functionally equivalent for this use case. **No issue.**

## 3. MEDIUM Issues

### 3.1 `summarizeValue` -- key sorting diverges from Python

**File:** `src/transcript/common.ts:113`
**Severity:** MEDIUM

```ts
text = JSON.stringify(value, Object.keys(value as Record<string, unknown>).sort())
```

Python's `json.dumps(value, sort_keys=True)` recursively sorts all keys in nested objects. `JSON.stringify` with a replacer array only includes the specified top-level keys, and does **not** recurse into nested objects' keys. This means nested objects will have their keys in insertion order, not sorted. While this only affects the `summarizeValue` output (cosmetic for tool event summaries), it's a behavioral divergence.
