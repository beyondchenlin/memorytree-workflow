## MemoryTree Workflow — 功能全景

### 一句话定义

为任意代码仓库添加 **持久化、Git 追踪的项目记忆层**，让 AI 编程助手（Claude Code、Codex、Gemini CLI）跨会话保留上下文。

---

### 核心架构

```
你的代码仓库/
├── AGENTS.md                    ← AI 行为规则（读取顺序、记忆规则、Git 规则）
└── Memory/
    ├── 01_goals/                ← 版本化的项目目标
    ├── 02_todos/                ← 与目标绑定的任务列表
    ├── 03_chat_logs/            ← 只追加的会话日志
    ├── 04_knowledge/            ← (可选) 持久化知识笔记、架构决策
    ├── 05_archive/              ← (可选) 已完成/退役的目标和待办
    └── 06_transcripts/          ← (可选) AI 对话历史归档
        ├── clean/               ←   清洗后的 Markdown 索引
        ├── manifests/           ←   导入元数据
        └── raw/                 ←   原始 transcript 镜像
```

全局数据存放在 `~/.memorytree/`（跨项目共享）：
```
~/.memorytree/
├── config.toml              ← 全局配置
├── alerts.json              ← 待处理通知
├── heartbeat.lock           ← 心跳单实例锁
├── logs/                    ← 心跳执行日志
└── transcripts/             ← 全局 transcript 归档
    ├── raw/                 ←   按 client/project/年/月 存放
    ├── clean/               ←   清洗后的 Markdown
    └── index/
        ├── sessions.jsonl   ←   会话元数据
        └── search.sqlite    ←   全文搜索索引
```

---

### 功能模块详解

#### 1. 自动检测与初始化

激活 Skill 时（运行 `/memorytree-workflow`），自动执行：

| 步骤 | 动作 |
|------|------|
| 检测状态 | 判断仓库是 `not-installed`、`partial` 还是 `installed` |
| 检查告警 | 读取 `~/.memorytree/alerts.json`，有通知则展示 |
| 未安装 | 运行 `init-memorytree.py` 脚手架，创建 `AGENTS.md` + `Memory/` |
| 部分安装 | 运行 `upgrade-memorytree.py`，补齐缺失文件，不覆盖已有策略 |
| 已安装 | 读取当前目标、待办、最新日志，汇报状态，询问下一步 |

**不需要人工确认就直接行动**——检测到状态后立即处理。

#### 2. 项目记忆管理（日常维护）

| 记忆类型 | 行为规则 |
|----------|----------|
| **Goals（目标）** | 只在用户明确确认需求/范围变化后才修改；版本化管理 |
| **Todos（待办）** | 与当前目标保持同步；进度变化时更新 |
| **Chat Logs（对话日志）** | 只追加，永不重写或删除历史记录 |
| **Knowledge（知识）** | 存放架构决策、产品想法、运维约束 |
| **Archive（归档）** | 过期内容从活跃文件移入归档，保持活跃文件精简 |

#### 3. 多客户端 Transcript 归档

支持从三个 AI 客户端导入对话历史：

| 客户端 | 数据源 |
|--------|--------|
| **Codex** | `~/.codex/sessions/**/rollout-*.jsonl` |
| **Claude Code** | `~/.claude/projects/<project>/*.jsonl` |
| **Gemini CLI** | `~/.gemini/` 下的 saved chat 或 checkpoint |

**关键原则**：
- 原始 transcript 作为证据不可篡改保留
- 清洗后的 Markdown 由**确定性代码**生成（不消耗模型 token）
- 当前仓库只保留属于本项目的 transcript；其他项目的只进全局归档
- 首次导入时询问用户：是否允许把原始 transcript 提交到仓库

#### 4. 后台心跳进程（合约规范，代码未实现）

## 场景还原

```

  └─ 用户讨论了很久，积累大量上下文
  └─ 问题没有解决，会话结束
```

## 这和现有心跳机制的关系

现有的心跳是**定时后台同步**（每 15 分钟），但这个场景需要的是 **按需即时同步**——用户刚关掉会话 A，马上开了会话 B，心跳可能还没跑过，最新的 transcript 还没进 Memory。

所以这不是心跳的职责，而是一个**新的交互行为**：

| 机制 | 触发方式 | 时效性 | 目的 |
|------|----------|--------|------|
| 心跳 | OS 调度器定时 | 延迟 ≤ 15 分钟 | 后台归档，持久化 |
| **新功能** | 用户在会话中请求 | 即时 | 上下文恢复，继续工作 |

## 我理解的完整流程

用户在新会话中说"看看我最近的聊天"时，Skill 应该：

**第一步：即时同步**
- 立刻调用 `discover-transcripts.py`（或类似的即时导入路径），不等心跳
- 只扫描 Claude Code 的本地 transcript 存储（`~/.claude/projects/<project>/`）
- 用当前项目目录做匹配，筛选出**属于本项目**的 transcript

**第二步：定位最近会话**
- 按时间戳排序，找到最新的那个 transcript
- 排除当前会话自身（刚开始的会话 B）
- 确认这就是"会话 A"

**第三步：总结输出**
- 读取该 transcript 的内容
- 提取关键信息：
  - 在讨论什么问题？
  - 尝试了哪些方案？
  - 走到了哪一步？
  - 什么没有解决？最后卡在哪里？
- 以结构化的方式输出给用户

**第四步：无缝继续**
- 用户看完总结后，直接说"继续"就能接上

## 几个需要想清楚的设计问题

### 1. 即时同步的执行方式

有两个选择：

- **A：Skill 直接调用 discover/import 脚本**——即时、可靠，但绕过了心跳的统一入口
- **B：调用 `memorytree-daemon run-once`**——复用心跳逻辑，但会同步所有项目，可能比"只同步当前项目"慢

我倾向 A，因为用户要的是**快速恢复上下文**，不需要全量同步。可以设计一个轻量的"单项目即时同步"路径。

### 2. 总结是否消耗 model token

现有规则说"transcript 清洗由确定性代码完成，不消耗 token"。但这里的**总结**不是清洗——它是一个有智能的摘要，必须用模型来做。

这和心跳的"清洗"职责不同：
- **清洗**：格式转换、去噪、生成索引 → 确定性代码
- **总结**：理解对话语义、提取未解决问题 → 必须用模型

所以这个功能本质上是**模型在交互会话中读取 transcript 并生成延续摘要**，符合现有规则"大模型只做 chat_log 摘要"的精神。

### 3. 和现有 chat_log 的关系

`Memory/03_chat_logs/` 已经有会话日志，但：
- 如果会话 A 没有正常结束（用户直接关窗口），chat_log 可能没写完
- chat_log 是会话**期间**写的摘要，不是**事后**的全面复盘
- transcript 才是完整的原始记录

所以这个功能的数据源应该是 **transcript（优先）+ chat_log（补充）**。

### 4. "最近的聊天"的范围

用户说"最近的聊天"可能指：
- 最近一次（最常见）
- 最近几次（"我这两天的讨论"）
- 最近关于某个主题的（"上次讨论部署的那次"）

第一版可以只处理"最近一次、本项目"，后续扩展。

### 5. 触发方式

两种可能：
- **被动触发**：用户明确说"看看最近的聊天"
- **主动触发**：Skill 激活时如果检测到有未同步的新 transcript，主动提示"发现你上次在这个项目有一个未完成的会话，要看看吗？"

主动触发体验更好，但需要在 On Activation 流程中增加检测步骤。

4. 可能的 **On Activation 主动检测**（不等用户问，主动发现未同步的会话）
