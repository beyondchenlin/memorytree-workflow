**Critical design decision: sql.js API**

sql.js initialization is async (`initSqlJs()` returns a Promise), but once initialized, the `SQL` object and `Database` operations are synchronous. The key async point is:

```ts
import initSqlJs, { type Database } from 'sql.js'

// One-time async init:
const SQL = await initSqlJs()

// Then synchronous operations:
const db = new SQL.Database(existingData) // or new SQL.Database()
db.run("CREATE TABLE IF NOT EXISTS ...")
db.run("INSERT ... ON CONFLICT ...", [...params])
const data = db.export()  // returns Uint8Array
writeFileSync(dbPath, Buffer.from(data))
```

Wait -- sql.js in Node.js mode loads the WASM file. The `Database` class methods (`run`, `exec`, `export`) are synchronous. The only async part is `initSqlJs()` which loads the WASM binary. So `upsertSearchIndex` needs to be async only for the initial load. We should cache the SQL module.

**Recommended approach**:
```ts
import initSqlJs from 'sql.js'
import { readFileSync, writeFileSync, existsSync } from 'node:fs'

let sqlPromise: Promise<SqlJsStatic> | null = null
function getSql(): Promise<SqlJsStatic> {
  if (!sqlPromise) {
    sqlPromise = initSqlJs()
  }
  return sqlPromise
}

export async function upsertSearchIndex(dbPath: string, manifest: ManifestEntry): Promise<void> {
  const SQL = await getSql()
  // Load existing DB or create new
  const db = existsSync(dbPath)
    ? new SQL.Database(readFileSync(dbPath))
    : new SQL.Database()
  try {
    db.run("PRAGMA journal_mode=WAL")
    db.run("CREATE TABLE IF NOT EXISTS transcripts (...)")
    db.run("INSERT ... ON CONFLICT ...", [manifest values])
    const data = db.export()
    writeFileSync(dbPath, Buffer.from(data))
  } finally {
    db.close()
  }
}
```

**Important note on WAL mode**: sql.js operates on in-memory copies of the database file. `PRAGMA journal_mode=WAL` has no practical effect since we load/export the entire DB as a byte array. However, we keep it for parity with the Python implementation and for future migration to native SQLite.

**Type definition for sql.js**: Since there are no bundled `.d.ts` files, we need to add `@types/sql.js` as a devDependency, or write a minimal declaration file at `src/types/sql.js.d.ts`. Check if `@types/sql.js` exists on npm. If not, create a minimal ambient declaration:
```ts
declare module 'sql.js' {
  interface Database {
    run(sql: string, params?: unknown[]): Database
    exec(sql: string): QueryExecResult[]
    export(): Uint8Array
    close(): void
  }
  interface QueryExecResult {
    columns: string[]
    values: unknown[][]
  }
  interface SqlJsStatic {
    Database: new (data?: ArrayLike<number>) => Database
  }
  export default function initSqlJs(config?: Record<string, unknown>): Promise<SqlJsStatic>
}
```

### Potential Challenges and Mitigations

1. **`normalizeTimestamp` precision differences**: Python's `datetime.fromisoformat` and JS `new Date()` have subtly different parsing behavior for edge cases (e.g., timezone offsets without colons, date-only strings). Mitigation: golden file tests with all input formats found in the Python test suite, plus additional edge cases.

2. **`slugify` non-ASCII handling**: Python's `encode("ascii","ignore")` drops all non-ASCII bytes. The TS regex `replace(/[^\x00-\x7F]/g, '')` is equivalent but operates on code points, not bytes. For the input domain (file paths and session IDs), this distinction does not matter. Mitigation: test with CJK characters, emoji, and combining marks.

3. **sql.js WASM loading in vitest**: sql.js needs to load its WASM binary. In vitest, this should work out of the box since vitest runs in Node.js. However, the WASM file path resolution may need configuration. If it fails, use `locateFile` config option in `initSqlJs()`:
   ```ts
   const SQL = await initSqlJs({
     locateFile: (file: string) => `node_modules/sql.js/dist/${file}`
   })
   ```

4. **`fs.globSync` not available in Node 20**: Node 20 does not have stable `fs.globSync`. Either add `fast-glob` as a dependency or implement recursive directory walking with `readdirSync({ recursive: true })` (stable since Node 18.17) followed by path pattern matching with `minimatch` or manual glob-to-regex conversion. The simplest approach: add `fast-glob` to `dependencies` in package.json.

5. **Windows path separators in tests**: All tests that construct paths must use `toPosixPath()` or forward slashes explicitly. Tests should pass on all 3 OS targets.

6. **Async propagation from db.ts**: Since `upsertSearchIndex` is async (due to sql.js WASM loading), `importTranscript` must also be async. This is the only async function in the transcript pipeline. All other functions remain synchronous.
