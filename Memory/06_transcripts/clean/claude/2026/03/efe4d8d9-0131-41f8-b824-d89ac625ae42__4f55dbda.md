## 先理解两种"定时做事"的方式

**方式一：你自己盯着（tmux 长驻进程）**

想象你雇了一个人，让他 24 小时坐在办公室里：
```
他坐在那 → 等 5 分钟 → 扫描一次 → 等 5 分钟 → 扫描一次 → ...
```
- 他一直在，一直占着一把椅子（内存）
- 你把办公室门关了（关终端），他还在（tmux 的作用）
- 但如果停电了（电脑重启），他就走了，你得再叫他来

**方式二：设个闹钟让系统叫他（OS 调度）**

你不雇人坐着，而是设个闹钟：
```
闹钟响 → 叫个人来 → 他扫描一次 → 做完走人 → 5分钟后闹钟再响 → 再叫人来 → ...
```
- 没人一直坐着，做完就走（不占内存）
- 闹钟是操作系统管的，停电重启后闹钟还在
- 每次来的是"新的一个人"，不会因为上一次出了问题而影响下一次

## "闹钟"是什么？

每个操作系统都自带定时任务工具，不需要安装任何东西：

| 你的系统 | 自带的"闹钟" | 你用过的类似东西 |
|---------|-------------|----------------|
| Linux | **cron** | 像手机定时闹钟，写一行就行 |
| macOS | **launchd** | 系统偏好设置里的"启动项" |
| Windows | **任务计划程序** | 控制面板里的"计划任务"，你可能见过 |

你用的是 Windows，所以就是**任务计划程序**——右键"我的电脑"→ 管理 → 就能看到它。Windows Update 就是靠它定时检查更新的。

## Python 脚本怎么运行？

你担心的问题：Python 是不是要一直在后台跑？

**不是。** 就像你双击一个 `.py` 文件：

```
系统闹钟响了
  → Windows 启动 python.exe
  → 跑 heartbeat.py（大约 2-3 秒）
  → 扫描、导入、commit
  → python.exe 自动退出（释放所有资源）
  → 什么都不占了
  → 5 分钟后闹钟再响，重复上面的过程
```

Python 只在那 2-3 秒内存在，其余时间完全不占任何资源。

## 对比一下

```
tmux 方案（雇人坐着）:
  python.exe 一直在后台 ──────────────────────→ 永远运行
  内存占用: 持续 ~30MB
  电脑重启: 丢失，要手动重启

OS 调度方案（设闹钟）:
  python.exe    python.exe    python.exe
  ──|2秒|──────────|2秒|──────────|2秒|──→
     ↑              ↑              ↑
   闹钟响          闹钟响          闹钟响
  内存占用: 99.9% 时间是 0
  电脑重启: 闹钟还在，自动继续
```

```
  │
  └→ 有内容（有变动）→ git add → git commit → 可选 git push
```

## 多项目怎么办？

你有 50 个项目，每个项目是独立的 Git 仓库。heartbeat 的流程是：

```
heartbeat.py 启动
  │

  │
  ├→ transcript-1 属于 项目A（D:\work\projectA）

  │   ├→ git add Memory/
  │   ├→ git commit -m "chore(memorytree): import transcript 2026-03-14"
  │   └→ ✓ 完成
  │
  ├→ transcript-2 属于 项目B（D:\work\projectB）
  │   ├→ 导入到 projectB/Memory/06_transcripts/
  │   ├→ cd D:\work\projectB
  │   ├→ git status → 有变动
  │   ├→ git add + commit
  │   └→ ✓ 完成
  │
  ├→ 项目C ~ 项目Z：没有新 transcript
  │   └→ 根本不会碰这些仓库（不 cd、不 status、不 commit）
  │
  └→ 退出
```

## 关于 push

这里有个关键决策：

```
commit = 保存到本地（安全，随时可做）
push   = 上传到远程（GitHub/Gitea）
```

| 策略 | 说明 | 建议 |
|------|------|------|
| **只 commit，不 push** | 变动存在本地，你下次手动 push | 最安全，适合默认行为 |
| **自动 push** | commit 后立即推到远程 | 方便，但要确保网络通、权限对 |
| **攒批 push** | 攒够 N 次 commit 或到整点才 push | 折中方案，减少网络请求 |

建议默认 **只 commit**，push 由用户手动或配置开启：

```toml
# memorytree-daemon.toml
auto_push = false          # 默认不自动 push
# auto_push = true         # 用户想自动就改成 true
```

## 安全保障

你可能担心：**会不会把不该提交的东西提交了？**

不会，因为：

1. **只动 Memory/ 目录** — `git add Memory/` 只暂存记忆文件，不碰代码
2. **遵守 .gitignore** — 如果 Memory/ 下有 .gitignore 规则，Git 自动跳过
3. **独立分支（可选）** — 你现有的 `git-policy.md` 已经设计了专用分支策略
4. **commit message 有标识** — 前缀 `chore(memorytree):` 一眼就能认出是自动提交

## 伪代码总结

```python
def auto_commit(project_path):
    os.chdir(project_path)
    
    # 检查有没有变动
    result = run("git status --porcelain Memory/")
    if not result.strip():
        return  # 没变动，跳过
    
    # 有变动，提交
    run("git add Memory/")
    run('git commit -m "chore(memorytree): import transcript 2026-03-14"')
    
    # 可选：自动 push
    if config.auto_push:
        run("git push")
```

就这么简单——**有变动就提，没变动就跳**。Git 本身的机制保证了这一点。
