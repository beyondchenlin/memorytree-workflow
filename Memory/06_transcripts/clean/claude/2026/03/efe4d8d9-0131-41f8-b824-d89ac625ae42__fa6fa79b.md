**Purpose**: Document the heartbeat architecture contract/spec. This is the single source of truth for how the background process works. All other files reference this instead of repeating details.

**Structure** (following the established Goal / Core Rules / sub-sections / Stop And Ask pattern from `execution-environment.md` and `transcript-archive.md`):

```
# Heartbeat Scheduling

Use this file when configuring or troubleshooting the MemoryTree background heartbeat.

## Goal
- One sentence: Keep transcript imports, cleaning, and git commits running without model tokens or user intervention.

## Core Rules
1. heartbeat.py runs once per invocation, then exits (stateless, idempotent)
2. OS-native scheduler calls it on interval (cron / launchd / Task Scheduler)
3. Global singleton: 1 machine = 1 scheduled task, scanning all registered projects
4. Client directories scanned: ~/.claude/projects/, ~/.codex/sessions/, ~/.gemini/
5. Each run: scan → incremental import → clean → git commit + push → exit
6. tmux loop mode (watch) exists only for development/debugging, not production

## Daemon CLI
- memorytree-daemon install — detect OS, register scheduled task, prompt user to confirm
- memorytree-daemon uninstall — remove scheduled task
- memorytree-daemon run-once — single manual execution (debugging)
- memorytree-daemon watch — tmux loop mode (dev/debug only)
- memorytree-daemon status — show scheduler registration state + last run result

## Execution Flow
1. Acquire lock file (~/.memorytree/heartbeat.lock) — skip if previous run still active
2. Load config from ~/.memorytree/config.toml
3. For each registered project: discover new transcripts, import, clean, stage Memory/**, commit, push (if auto_push and remote exists)
4. On push failure: log warning, increment failure counter, write to alerts.json after N consecutive failures
5. Release lock, exit

## Sensitive Info Scanning
- During transcript cleaning, scan for common patterns: API keys, passwords, tokens, connection strings
- If found: log warning with file path and line, do NOT auto-delete (avoid false positives)
- User can review warnings in heartbeat log

## Git Remote Pre-Check
- Before push, verify remote exists (git remote -v)
- No remote → skip push, log warning, write alert for next Skill activation

## First Install
- init-memorytree.py completion prompts user to run `memorytree-daemon install`
- install command requires explicit user confirmation (never silently register)

## Stop And Ask
Stop and ask before any of these:
1. Silently registering a scheduled task without user confirmation
2. Pushing to a repository that has no remote configured
3. Overwriting an existing scheduled task registration from another MemoryTree version
```

**Structure**:

```
# Global Configuration

Use this file when reading or modifying MemoryTree's user-level global configuration.

## Goal
- One sentence: Provide a single user-level location for MemoryTree's global state, configuration, and cross-project data.

## Directory Layout

~/.memorytree/
  config.toml          # Global configuration (heartbeat interval, watch dirs, projects, auto_push)
  alerts.json          # Accumulated error notifications for Skill activation display
  heartbeat.lock       # Lock file for single-instance enforcement
  logs/
    heartbeat.log      # Daily-rotated heartbeat execution log
  transcripts/         # Global transcript archive (already documented in transcript-archive.md)
    raw/
    clean/
    index/

Note: Path roots: Windows %USERPROFILE%, macOS/Linux $HOME.

## config.toml Schema
- heartbeat_interval: integer, seconds between runs (default: 300)
- watch_dirs: list of strings, client transcript directories to scan
  - default: ["~/.claude/projects/", "~/.codex/sessions/", "~/.gemini/"]
- projects: list of objects, registered project roots
  - each entry: { root = "/path/to/repo" }
  - auto-discovered on first Skill activation per repo, or manual add
- auto_push: boolean (default: true) — push Memory commits to remote automatically
  - Rationale: memory is a user asset; local-only storage is an unrecoverable loss risk
  - First install prompts user to confirm; after confirmation, silent execution
- log_level: string, one of "debug", "info", "warning", "error" (default: "info")

## alerts.json
- Array of alert objects: { timestamp, project, type, message, count }
- Written by heartbeat when errors accumulate (e.g., push fails N consecutive times)
- Read and displayed by Skill on activation, then cleared after display
- Types: push_failed, no_remote, lock_timeout, sensitive_info_detected

## Project Registration
- Automatic: Skill activation in a new repo adds it to projects list
- Manual: user edits config.toml directly
- Unregistered repos are skipped by heartbeat

## Stop And Ask
Stop and ask before any of these:
1. Modifying config.toml on behalf of the user without confirmation
2. Clearing alerts.json without displaying alerts to the user first
3. Adding a project to the registered list when the repo has no Memory/ directory
```

**Add a new section "Auto Push" after "Default MemoryTree Flow" (after line 43) and before "Commit Guidance"**:

```
## Auto Push

Auto push is controlled by `auto_push` in `~/.memorytree/config.toml` (default: `true`).

When `auto_push` is `true`:

1. The heartbeat process pushes MemoryTree-owned commits to the current branch's remote automatically after each commit.
2. Push targets the current branch. The heartbeat does not create branches or PRs; it commits directly to the branch the project is on.
3. Before push, verify the remote exists. If no remote is configured, skip push and write an alert to `~/.memorytree/alerts.json`.
4. On push failure (network error, auth failure, rejected push), log the error, increment failure counter, and retry on the next heartbeat run. After N consecutive failures, write an alert.
5. First install prompts the user to confirm auto_push behavior. After confirmation, execution is silent.

When `auto_push` is `false`:

1. The heartbeat still commits locally but does not push.
2. The user is responsible for pushing MemoryTree changes manually.

Rationale: Memory is a user asset. Local-only storage means a single disk failure loses all memory permanently. Auto push is the safe default.
```

**Modify "Stop And Ask" section**: Add item 8:
