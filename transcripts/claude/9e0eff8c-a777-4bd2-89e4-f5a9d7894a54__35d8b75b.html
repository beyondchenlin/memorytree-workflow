# Second-Round Security and Consistency Review

## 1. SECURITY REVIEW

### 1.1 SQL Injection in src/transcript/db.ts

Rating: OK (No Issue)

db.ts:84-90 builds the UPSERT statement with ? placeholders and passes values via a params array (line 126: db.run(UPSERT_SQL, params)). The column names in ALL_COLUMNS are static string literals defined at module scope (lines 30-52), not derived from user input. The CREATE_TABLE_SQL is also a static string. Parameters are properly bound. No injection vector exists.

### 1.2 Command Injection in src/utils/exec.ts

Rating: OK (No Issue)

exec.ts:23 uses execFileSync(command, [...args]), which is the safe variant that does not invoke a shell. Arguments are passed as an array, preventing shell metacharacter injection. The command parameter is always a hardcoded string ('git', 'tasklist', 'crontab', 'schtasks', 'launchctl'), never user-supplied.

Rating: MEDIUM -- One exception at src/cli/cmd-daemon.ts:139:

execCommand('bash', ['-c', `crontab "${tmpFile}"`])

This uses bash -c which re-introduces shell interpretation. If tmpFile contained shell metacharacters (e.g., if the homedir contained $(...) or backticks), this could be exploited. The tmpFile path is derived from homedir() which is OS-controlled, making exploitation unlikely but not impossible. The same pattern appears at line 157.

File: D:\demo1\memorytree-workflow\src\cli\cmd-daemon.ts lines 139, 157

Recommendation: Replace bash -c with a direct crontab call using stdin piping, or use execFileSync('crontab', [tmpFile]) directly.

- scaffold.ts:102-109 (resolveTemplateDir) uses join(skillRoot, 'assets', 'templates', localeName). If localeName contained ../.., it could escape the templates directory. The normalizeLocale function (locale.ts:22-41) normalizes to known values or passes through unknown values unchanged. A locale like ../../etc would pass through as-is.

### 1.4 Sensitive Data in Logs

### 1.5 Lock File Race Conditions (TOCTOU) in lock.ts

Rating: LOW

The acquireLock() function at lock.ts:42-83 has a two-phase approach:
1. Phase 1 (lines 46-69): Checks if lock file exists, reads PID, checks if process is alive, removes stale lock.
2. Phase 2 (lines 72-81): Creates lock file with O_CREAT | O_EXCL.

There is a TOCTOU window between removing the stale lock (line 61 or 67) and creating the new lock (line 73). Two processes could both detect the stale lock, both remove it, and then both attempt O_CREAT | O_EXCL. However, only one will succeed because O_EXCL is atomic at the filesystem level. The losing process falls through to the catch block (line 77) and returns false. This is correctly handled.

The comment at line 71 ("Use exclusive create to prevent TOCTOU race") is accurate. The design is sound.

### 1.6 Config TOML Injection

Rating: MEDIUM

config.ts:82 constructs TOML by string interpolation:

const items = cfg.watch_dirs.map(d => `"${d}"`).join(', ')

If a watch_dirs entry contains a double quote or backslash, the generated TOML will be malformed or could inject additional TOML keys. Similarly, config.ts:91-92:

lines.push(`path = "${project.path}"`)
lines.push(`name = "${project.name}"`)

Neither project.path nor project.name are escaped for TOML string syntax.

File: D:\demo1\memorytree-workflow\src\heartbeat\config.ts lines 82, 91-92

Recommendation: Escape double quotes and backslashes in TOML string values before interpolation, or use a TOML serialization library.

### 1.7 Launchd Plist XML Injection

Rating: MEDIUM

cmd-daemon.ts:196-217 builds a plist XML string by interpolating scriptPath and logDir values directly into XML without escaping <, >, &, or ". If the script path or log directory contains XML special characters, the plist will be malformed or could inject arbitrary XML elements.

File: D:\demo1\memorytree-workflow\src\cli\cmd-daemon.ts lines 196-217

Recommendation: XML-escape the interpolated values, or use a proper plist builder.
