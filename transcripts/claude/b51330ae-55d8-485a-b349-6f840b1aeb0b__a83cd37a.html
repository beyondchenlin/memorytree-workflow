## npm Package Configuration Review — @openmnemo Monorepo

---

### CRITICAL

1. @openmnemo/report — @anthropic-ai/sdk should be a peer dependency, not a regular dependency

File: D:/demo1/openmnemo/packages/report/package.json

@anthropic-ai/sdk is listed as a regular dependency and loaded via a dynamic await import("@anthropic-ai/sdk") only when AI summarization is active. Because it is a regular dep, every consumer who installs @openmnemo/report (or @openmnemo/cli, which depends on report) will pull in the entire Anthropic SDK (~several MB, complex native bindings on some platforms) whether they ever use AI summaries or not. Furthermore, the CLI already passes --no-ai as a default-off flag, meaning most users never trigger the code path. The correct pattern is: list it as peerDependencies with peerDependenciesMeta.optional: true, keep the dynamic import, and let consumers who want AI summaries install the SDK themselves. As-is, this inflates install size for all users and can cause version conflicts if a consuming project already pins a different version of the SDK.

2. @openmnemo/cli — program.name('memorytree') and program.version('0.1.0') do not match the package

File: D:/demo1/openmnemo/packages/cli/src/main.ts lines 11 and 13

The CLI binary is registered as openmnemo in package.json's bin field and the package is @openmnemo/cli at version 0.2.0. But the source hardcodes .name('memorytree') and .version('0.1.0'). A user who runs openmnemo --help sees "memorytree", and openmnemo --version prints 0.1.0. This is a direct user-facing inconsistency and will cause confusion. The version should be read from package.json at runtime (e.g., via import { createRequire } from 'module' or by passing it as a build-time constant via tsup's define), and the program name must be openmnemo.

3. @openmnemo/types — dist/index.js is 0 bytes, yet "main" points to it

D:/demo1/openmnemo/packages/types/dist/index.js is confirmed 0 bytes. This is expected because tsup strips type-only exports to nothing in the JS output. Node.js will import an empty file without error, so runtime usage is fine. However, several tooling scenarios break: bundlers that expect main to return something (webpack's sideEffects analysis), tools that check whether a package entry resolves to a non-empty file, and any consumer who tries require.resolve('@openmnemo/types') as an existence check. More critically, there is no exports field, so package.json with "type": "module" + "main": "./dist/index.js" is the only resolution path. The correct fix for a pure-types package is either: (a) add an exports field with a "types" condition only, removing the misleading "main", or (b) add a trivial // types only comment line in dist/index.js so the 0-byte file is intentional and documented. The "types" field pointing to dist/index.d.ts is fine — that file has content (1178 bytes).

### HIGH

4. No exports field on any package — Node.js subpath and conditional export resolution is broken

Files: all five package.json files under packages/

None of the packages define an "exports" field. Without "exports", Node.js (v12+) and modern bundlers use "main" as a fallback, which works for the single entry point. But this means:
- Consumers cannot use "imports" conditions (e.g., "types", "import", "default") that TypeScript 5.x expects when moduleResolution: "bundler" or "node16" is set.
- Deep imports like import { something } from '@openmnemo/core/transcript/parse' are not blocked, allowing accidental coupling to internal paths.
- TypeScript with "moduleResolution": "node16" or "nodenext" will fail to find types unless "exports" has a "types" condition.

Minimum fix: add "exports": { ".": { "import": "./dist/index.js", "types": "./dist/index.d.ts" } } to all packages. For @openmnemo/types, the entry should only have "types".

5. @openmnemo/sync and @openmnemo/report have no credible standalone user story, yet are published as public packages

Both packages' description fields describe internal subsystems ("heartbeat daemon", "report generation"). There is no install-and-use story independent of the CLI. Publishing them publicly creates a fragmented surface: users who stumble upon npm i @openmnemo/sync get a daemon management API with no documentation or stable interface promise. The recommendation is to mark these "private": true or add "publishConfig": { "access": "restricted" } unless there is a deliberate extension API intent. If they must be public (for the programmatic API use case), they each need a CHANGELOG and clear semver commitment.

6. No engines field in any individual package — consumers on Node 18 may silently get broken behavior

File: root package.json has "engines": { "node": ">=20" }, but none of the published packages declare this. When a user installs @openmnemo/cli on Node 18, npm will not warn them. The CLI uses sql.js, dynamic imports with top-level await patterns, and commander v13 — all of which require Node 18+, but some patterns (notably import.meta.resolve usage if present) require Node 20+. Each published package should declare "engines": { "node": ">=20" } so npm warns users at install time.

7. Publish workflow skips tests

File: D:/demo1/openmnemo/.github/workflows/publish.yml

The publish workflow runs only pnpm install && pnpm build && pnpm publish. It does not run pnpm test or pnpm typecheck before publishing. The CI workflow does run tests, but CI only triggers on push to main and pull_request — not on tag pushes. A tag push triggers only the publish workflow, which has no test gate. A broken build could be published. Add pnpm test && pnpm typecheck before the publish step, or add a needs: ci dependency on a separate test job.

### MEDIUM

8. workspace:* resolves to exact version at publish time — correct behavior, but no version range strategy

pnpm 9 correctly rewrites workspace: to "0.2.0" (exact) when publishing. Since all packages are at 0.2.0, the published @openmnemo/cli@0.2.0 will declare "@openmnemo/core": "0.2.0" as an exact dep. This means if @openmnemo/core ships a patch fix as 0.2.1, consumers who install @openmnemo/cli@0.2.0 will not get the fix unless CLI is also republished. Consider using workspace:^ instead of workspace: in package manifests, which pnpm rewrites to "^0.2.0", allowing patch/minor updates to propagate independently. This matters most for @openmnemo/types and @openmnemo/core since they are the most likely to receive fixes.
