## Code Review Report: Heartbeat System and Session Continuity

### CRITICAL

1. memorytree_daemon.py lines 339-343 -- _python_path() returns a multi-word string used unsplit in subprocess.run() list arguments

def _python_path() -> str:
    uv = shutil.which("uv")
    if uv:
        return f"{uv} run python"
    return sys.executable

This returns e.g. "/home/user/.local/bin/uv run python" as a single string. It is then used in:

# line 119
result = subprocess.run([python, str(heartbeat_script)], ...)
# line 133
subprocess.run([python, str(heartbeat_script)], ...)

When uv is found, this becomes ["/usr/bin/uv run python", "heartbeat.py"] -- the OS will try to find a literal executable named "uv run python" (with spaces) and fail with FileNotFoundError. The string needs to be split into ["uv", "run", "python"] before being placed in the list.

This also affects _install_cron() (line 187), _install_launchd() (line 243), and _install_schtasks() (line 299) where python is interpolated into command strings or plist XML. The cron/plist/schtasks cases happen to work for the shell-based paths because the whole command is interpreted by cron's shell or launchd, but subprocess.run() with a list will absolutely fail.

2. _lock_utils.py lines 19-35 -- TOCTOU race condition in acquire_lock()

Between checking path.exists() on line 19 and writing the PID on line 32, another process can create the lock file. Two concurrent heartbeat instances starting simultaneously can both see "no lock file" and both write their PID, resulting in two heartbeats running simultaneously. This matters because the heartbeat does git commit/push operations where concurrent execution can cause conflicts.

A proper fix would use open(path, 'x') (exclusive create) or fcntl.flock / msvcrt.locking for atomic lock acquisition.

3. recall-session.py lines 138-151 -- SQLite connection not closed on exception path

conn = sqlite3.connect(db_path)
conn.row_factory = sqlite3.Row
cursor = conn.execute(...)
rows = cursor.fetchall()
conn.close()

If conn.execute() or cursor.fetchall() raises an exception that is not sqlite3.Error or OSError (e.g. MemoryError, KeyboardInterrupt), conn.close() is skipped. Should use a with context manager or try/finally. Additionally, even on the handled exception path (line 152-153), the conn that was successfully opened is never closed before falling through to the JSONL fallback.

4. _alert_utils.py lines 41-46 -- In-place mutation of alert dicts

        alert["timestamp"] = now
        alert["message"] = message

The project's coding style explicitly requires immutable patterns. The alert dict (read from disk) is mutated in place. A new dict should be created and substituted into the list, or a new list should be built.

5. memorytree_daemon.py line 187 -- Shell injection via interpolated paths in cron line

cron_line = f"*/{minutes} * * * * cd {heartbeat_script.parent} && {python} {heartbeat_script} >> ..."

The heartbeat_script.parent and python values are interpolated directly into a cron entry that is passed to crontab -. If any of these paths contain shell metacharacters (spaces, semicolons, backticks, etc.), this can break or be exploited. The paths should be shell-quoted using shlex.quote().

The same issue exists in the launchd plist (line 235-256), though XML escaping is less exploitable. And in _install_schtasks() (line 299), the f'"{python}" "{heartbeat_script}"' approach partially mitigates this but would still fail if the path contains double quotes.

6. heartbeat.py lines 22-30 -- Import of _transcript_utils will fail at runtime

heartbeat.py does from _transcript_utils import ... using bare module names. There is no __init__.py in the scripts/ directory (confirmed by glob), and these are not installed packages. This import style only works if the scripts are invoked with cwd set to the scripts/ directory, or if scripts/ is on sys.path.

The existing scripts (discover-transcripts.py, import-transcripts.py) use the same pattern and rely on cwd=scripts/ at invocation time. The daemon's cmd_run_once() (line 119) passes cwd=heartbeat_script.parent which covers this, and cmd_watch() (line 133) does too. But _install_cron() line 187 uses cd {heartbeat_script.parent} && which is correct. So this works by convention, but it's fragile -- any invocation without setting cwd will fail. This is consistent with existing scripts so marking as HIGH rather than CRITICAL.

7. recall-session.py line 217 -- Overly broad path matching allows false positives

def _cwd_matches(cwd: str, resolved_root: str) -> bool:
    ...
    return cwd_resolved == resolved_root or resolved_root in cwd_resolved

The resolved_root in cwd_resolved check is a substring match, not a path hierarchy check. For example, if resolved_root is /home/user/app and cwd_resolved is /home/user/app-backup/subdir, this would return True as a false positive. Should use cwd_resolved.startswith(resolved_root + "/") or proper Path.parents logic.

### MEDIUM

8. _config_utils.py lines 115-134 -- save_config does not escape special characters in TOML values

lines.append(f'path = "{project.path}"')

If project.path contains backslashes (common on Windows, e.g. C:\Users\...) or double quotes, the resulting TOML will be malformed. Backslashes in TOML double-quoted strings are escape characters. Windows paths will be corrupted on round-trip (e.g., C:\Users\new becomes C:<TAB>Users<newline>ew when the TOML is re-read). The values should use TOML literal strings (single quotes) or properly escape backslashes.

Note: register_project() stores paths using .as_posix() (forward slashes), which avoids this for projects registered through that function. But if a user manually edits config.toml with Windows paths or if cfg.watch_dirs contains backslash paths, the round-trip through save_config + load_config will corrupt them.

9. _alert_utils.py -- No file locking on alerts.json read-modify-write

write_alert() reads the file, modifies, and writes back without any lock. If two heartbeat processes (or a heartbeat + recall-session) run concurrently, one can overwrite the other's alert. The lock file only protects heartbeat, not recall-session.py.

10. memorytree_daemon.py line 71-85 -- Config override creates new Config without validating the interval

The user-provided args.interval is stored directly without validation through _is_valid_interval(). An invalid value like "banana" would be saved to config.toml via save_config() and then interval_to_seconds() would silently fall back to the default. Not a crash, but unexpected behavior.

11. heartbeat.py line 132 -- _scan_sensitive has no type annotation for parsed parameter

The parsed parameter lacks a type annotation. Should be parsed: ParsedTranscript.

12. recall-session.py lines 99-121 -- Silently swallows all exceptions from parse and import

Both the parse and import error handlers use bare except Exception: continue with no logging. When invoked standalone (not from heartbeat), there is no way to know why transcripts were skipped. 15. _config_utils.py line 102-106 -- interval_to_seconds with "0m" input causes infinite recursion

If DEFAULT_HEARTBEAT_INTERVAL were ever changed to "0m" or an invalid value, this would infinitely recurse. Currently safe because the default is "5m", but the code is fragile. Better to validate the default at module level or use a hardcoded fallback.

When a second heartbeat instance finds the lock held, it writes an alert. Since OS schedulers can legitimately overlap (e.g., previous run took longer than the interval), this generates repeated alerts for normal operation. This may be intentional but could be distracting.
