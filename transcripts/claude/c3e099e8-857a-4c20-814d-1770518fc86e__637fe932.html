什么是 npm token

npm token 是 npm（Node.js 包管理平台）颁发的访问凭证，用于在命令行或 CI/CD 流水线中以你的账户身份执行操作，最常见的用途是发布包（npm publish）。

它的工作方式：
- 你登录 npm 网站，在账户设置里生成一个 token（一串随机字符串）
- 把这个 token 配置在 CI/CD（比如 GitHub Actions secrets 里的 NPM_TOKEN）
- 流水线运行 npm publish 时，用这个 token 代替密码进行身份验证

为什么之前要求轮换？

因为 token 曾经在 session 对话记录里出现过（比如命令输出或配置内容被记录进了对话文件里）。一旦 token 泄露，任何拿到它的人都可以：
- 以你的名义发布新版本的包
- 覆盖已有版本（如果是 automation token）
- 污染供应链

轮换的操作：
1. 去 [npmjs.com](https://www.npmjs.com) → Account Settings → Access Tokens → 删除旧 token，生成新 token
2. 在 GitHub 仓库 Settings → Secrets → 把 NPM_TOKEN 的值替换为新 token

如果你已经做了，在 todo 里把那行勾掉就好。

| 层级 | 范围 | 用途 |
|------|------|------|
| Conversation | 当前轮次 | 对话上下文 |
| Session | 短期任务 | 自动过期，用 session_id 隔离 |

### 记忆写入流程

1. memory.add() 调用 → LLM 提取关键事实/决策/偏好
2. 与已有记忆做冲突检测（去重 + 新信息覆盖旧信息）

## 一、向量数据库选型

sqlite-vec 是最适合 OpenMnemo 的选择，原因是零额外进程、数据文件可以 git 管理、和现有 db.ts 无缝集成。

npm install sqlite-vec

原理：给现有 SQLite 加一个扩展，新增一种列类型 float32[N]，支持 KNN 近邻查询。

-- 建向量表
CREATE VIRTUAL TABLE vec_sessions USING vec0(
  session_id TEXT,
  embedding float32[1536]   -- OpenAI ada-002 维度
);

-- 插入
INSERT INTO vec_sessions VALUES ('sess_abc', '[0.1, 0.2, ...]');

-- 检索最近邻 top-5
SELECT session_id, distance
FROM vec_sessions
WHERE embedding MATCH '[0.3, 0.1, ...]'
ORDER BY distance LIMIT 5;

现有 FTS4 表做关键词匹配，vec_sessions 做语义匹配，两个结果合并用 RRF（Reciprocal Rank Fusion）算法重排序，这就是"混合检索"。

## 三、LLM 提取层详解

这是整个记忆体系的"大脑"，在 heartbeat 同步时触发。

新对话导入
    ↓
Haiku 读对话全文（便宜，~$0.0001/次）
    ↓
提取三类信息：
  1. 关键事实   → "决定用 sqlite-vec 做向量检索"
  2. 实体       → [SQLite, openmnemo, FTS, 向量检索]
  3. 实体关系   → SQLite --用于--> openmnemo
    ↓
┌──────────────────────────────────────┐
│  sqlite-vec: 对"关键事实"做 embedding  │  ← 语义检索用
│  Kuzu: 写入实体节点 + 关系边           │  ← 关系追踪用
│  SQLite FTS: 写入对话全文              │  ← 关键词检索用
└──────────────────────────────────────┘

Haiku 的提取 prompt 大概长这样：

从以下对话中提取：
1. 关键决策/结论（1-3句话）
2. 涉及的技术/项目/人名实体
3. 实体之间的关系（三元组格式：主体-关系-客体）

对话内容：
{conversation_text}

返回 JSON：
{
  "facts": ["决定用 sqlite-vec", "FTS4 只索引了 title/cwd/branch"],
  "entities": [{"name": "sqlite-vec", "type": "technology"}, ...],
  "relations": [{"from": "sqlite-vec", "relation": "用于", "to": "openmnemo"}, ...]
}

## 四、双存储混合检索流程

用户查询："我上周讨论过什么向量数据库方案"
    ↓
并行执行三路检索：
  ① FTS4:    MATCH "向量数据库"  → [session_A rank:1, session_C rank:3]
  ② sqlite-vec: KNN embedding   → [session_A rank:2, session_B rank:1]
  ③ Kuzu:    MATCH (e:Entity {type:'technology'})-[:MENTIONED_IN]->(s)
             WHERE e.name CONTAINS '向量' → [session_A, session_D]
    ↓
RRF 合并排序（每路结果按 1/(k+rank) 打分求和）
    ↓
返回 top-5 session，附带摘要 + 关联 commit

(Concept)-[:DEPENDS_ON]->(Technology)    # 向量检索依赖 sqlite-vec
(Concept)-[:EVOLVED_FROM]->(Concept)     # 方案 v2 从方案 v1 演化而来
(Project)-[:USES]->(Technology)          # openmnemo 使用 TypeScript
