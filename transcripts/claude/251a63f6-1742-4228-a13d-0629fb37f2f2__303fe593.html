### Issue 3 — HIGH: rss.ts — new Date('invalid') does not throw; the try/catch in toRfc2822 is ineffective

File: src/report/render/rss.ts lines 82–88

function toRfc2822(iso: string): string {
  try {
    return new Date(iso).toUTCString()
  } catch {
    return new Date().toUTCString()
  }
}

In Node.js, new Date('invalid-string') does not throw an exception — it returns an Invalid Date object. Calling .toUTCString() on an Invalid Date returns the string "Invalid Date", which is not valid RFC 2822. The catch block is therefore unreachable dead code and provides no protection.

This means any manifest with a malformed started_at will silently produce <pubDate>Invalid Date</pubDate> in the RSS feed, which RSS validators and feed readers will reject at the item level.

Fix: Check isNaN(date.getTime()) before calling .toUTCString():

function toRfc2822(iso: string): string {
  const d = new Date(iso)
  return isNaN(d.getTime()) ? new Date().toUTCString() : d.toUTCString()
}

The rss.test.ts suite has no test for an invalid/malformed date string, leaving this path completely untested.

### Issue 4 — HIGH: toc.ts — injectHeadingIds regex [^<]* fails to match headings with inline HTML (bold/italic)

File: src/report/render/toc.ts line 93

return html.replace(/<h([23])>([^<]*)<\/h[23]>/g, (full, levelStr, innerHtml) => {

The regex ([^<]*) captures only text content with no < characters. However, markdownToHtml applies inline formatting to heading text via applyInline(), which turns bold into <strong>bold</strong> and ` code into <code>code</code>`. Any heading containing bold, italic, or inline code will produce HTML like:

<h2><strong>Overview</strong></h2>

This heading will not be matched by the regex (the < in <strong> breaks [^<]*), so no id attribute gets injected, and the TOC anchor link becomes a dead link.

The toc.test.ts tests only use plain-text headings ('Section One', 'Sub Section'), so this path is not tested. The injectHeadingIds test case "leaves unmatched headings unchanged" (line 95–100) does not test the inline-HTML case.

Fix: Widen the capture group from [^<] to [^]? (or [\s\S]*?) to allow any inner HTML:

return html.replace(/<h([23])>([\s\S]*?)<\/h[23]>/g, ...)

The matching logic comparing escHtml(e.text) to innerHtml.trim() also needs to account for stripped HTML tags when comparing. This is a deeper design issue: the plain-text extracted by extractToc (which reads raw markdown) will not match the HTML-rendered text. A robust fix would be to strip HTML tags from innerHtml before comparison.

### Issue 9 — MEDIUM: layout.ts — htmlShell renders reader-mode button outside content area unconditionally when readerMode is not explicitly false

### Issue 10 — MEDIUM: breadcrumb.ts — non-last item without a url gets no link and no aria-current, effectively becoming a dead span

File: src/report/render/breadcrumb.ts lines 31–36

const parts = items.map((item, i) => {
  const isLast = i === items.length - 1
  if (isLast || !item.url) {
    return `<span class="breadcrumb-item" ${isLast ? 'aria-current="page"' : ''}>${escHtml(item.label)}</span>`
  }
  return `<a class="breadcrumb-item" href="${escHtml(item.url)}">${escHtml(item.label)}</a>`
})

If a non-last item has no url (e.g., an intermediate crumb that serves as a category label), it renders as a <span> without aria-current, which is correct. However, the space before the conditional aria-current attribute produces <span class="breadcrumb-item" > (trailing space before >) when isLast is false and !item.url is true. While not a functional bug in most browsers, this is malformed HTML by strict standards.

Fix: Change to:

const ariaCurrent = isLast ? ' aria-current="page"' : ''
return `<span class="breadcrumb-item"${ariaCurrent}>${escHtml(item.label)}</span>`
