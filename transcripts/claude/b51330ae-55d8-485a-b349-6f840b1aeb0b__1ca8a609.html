File: /packages/core/src/transcript/parse.ts (lines 407-429)
Issue: Header regex patterns fail on Windows (CRLF) line endings.

When content.split('\n') is called on a file with CRLF endings (\r\n), the resulting lines retain the \r character at the end:
- Line: "Title: 对话标题\r"
- Regex: /^Title:\s*(.+)$/
- Result: NO MATCH (the $ cannot match when \r is at the end and .+ cannot include \r)

Affected patterns:
- Line 407: /^Title:\s*(.+)$/ - FAILS with CRLF
- Line 413: /^URL:\s*(.+)$/ - FAILS with CRLF
- Line 422: /^Created:\s*(.+)$/ - FAILS with CRLF

Turn header pattern works: Line 390's /^(User|AI):\s\[(\d{4}-\d{2}-\d{2}\s+\d{2}:\d{2}:\d{2})\]\s$/ succeeds because \s*$ at the end matches the \r.

Impact: A Windows user exporting a Doubao transcript to TXT with CRLF endings will have parsing failures for:
- Title (reverts to filename stem)
- URL parsing (session_id derivation fails)
- Created date parsing (uses file mtime instead)
