## 逐层分析

### L1 网络身份 — 两边都有短板

MediaCrawler 有代理池但没有 TLS 指纹管理。Pinchtab 完全没有代理。而现在抖音、小红书都在用 JA3/JA4 指纹检测 — 你用 Go 的 net/http 或 Python 的 httpx 发请求，TLS 握手特征跟真实 Chrome 完全不同，代理换再多 IP 也会被识别。

需要新增：
- TLS 指纹伪装（用 [utls](https://github.com/refraction-networking/utls) 或在 Pinchtab 的 chromedp 层面走真实浏览器 TLS）
- HTTP/2 指纹一致性（SETTINGS frame 顺序、WINDOW_UPDATE 大小）
- 代理池统一管理，从 MediaCrawler 移植 proxy_ip_pool.py 逻辑到 Go 侧，作为独立 sidecar

### L2 浏览器指纹 — Pinchtab 胜出，但缺一致性

Pinchtab 的 canvas 噪声、WebGL、字体、硬件伪装已经很好。但现在的问题是：每次轮换指纹后，各维度之间不一致。比如 UA 说 Mac + Safari，但 navigator.platform 返回 Win32，WebGL vendor 返回 Intel — 这种矛盾本身就是检测信号。

需要新增：
- 身份档案（Identity Profile）：一个 profile 包含一致的 OS/Browser/Screen/GPU/Language/Timezone 组合
- 预定义 50+ 真实设备档案，从真机采集
- 指纹轮换时整体切换，不是单维度随机

### L3 行为指纹 — Pinchtab 有基础，但不够深

Pinchtab 的 Bézier 鼠标、打字模拟已经不错。但平台现在检测的是更高层的行为模式：

- 你打开页面后直接定位到目标元素 → 机器人
- 你从不滚动页面 → 机器人
- 你每次停留时间完全一样 → 机器人

需要新增：
- 页面浏览模拟：先滚动浏览，随机停留，再定位目标
- 停留时间分布：正态分布，不是固定值
- 滚动行为：惯性滚动、回滚、速度变化
- 鼠标闲置轨迹：不操作时也有微小移动

### L4 平台签名 — MediaCrawler 只覆盖了 XHS

这是最关键的一层。7 个平台各有不同的签名机制：

| 平台 | 签名机制 | MediaCrawler 现状 | 对抗策略 |
|------|---------|-------------------|---------|
| 小红书 | x-s, x-t, x-s-common | 有 (playwright_sign + xhs_sign) | 保留，通过 Pinchtab 浏览器执行 |
| 抖音 | a_bogus, msToken, ttwid, _signature | 无 | 必须浏览器内执行 JS，不可逆向 |
| B站 | wbi 签名 (img_key + sub_key) | 无 | 可纯算法实现，定期更新 key |
| 快手 | did + captcha | 无 | 设备 ID 绑定 + 验证码服务 |
| 微博 | cookie 为主 | 基础 cookie 管理 | 增强 cookie 生命周期管理 |
| 贴吧 | tbs token | 基础 | 简单，保持 session 即可 |
| 知乎 | x-zse-96 (基于 x-zst-81) | 无 | 浏览器内执行，类似抖音 |

核心思路：放弃纯算法逆向签名，统一走 Pinchtab 浏览器执行。原因：
1. 抖音的 a_bogus 每隔几周更新混淆，纯逆向维护成本极高
2. Pinchtab 本身就是真实浏览器环境，直接在页面上下文里调用平台自己的签名函数
3. XHS 的 window.mnsv2 方案（MediaCrawler 已有）正好可以推广到其他平台

### L5 验证码应答 — MediaCrawler 只有滑块

MediaCrawler 的 OpenCV 滑块识别 + easing 轨迹生成是好的基础，但平台验证码类型远不止滑块：

| 类型 | 平台 | 现状 | 方案 |
|------|------|------|------|
| 滑块拼图 | XHS, 抖音 | MediaCrawler 有 | 保留，接入 Pinchtab 的 human.MouseMove |
| 点选文字 | 抖音, B站 | 无 | 接第三方打码 API 或本地 OCR |
| 旋转图片 | 快手 | 无 | 图像相似度匹配 |
| 行为验证 | 全平台 | 无 | L3 行为层覆盖 |
| 短信验证 | 全平台 | 无 | 接码平台 API（手动兜底） |

## 关键设计决策

1. Pinchtab 作为唯一浏览器引擎 — MediaCrawler 的 Playwright 层完全替换掉，签名、Cookie、页面交互全部走 Pinchtab API

2. MediaCrawler 降级为 "平台知识库" — 保留它的平台特定逻辑（签名算法、Cookie 字段、API endpoint），但执行环境换成 Pinchtab

3. Identity Manager 是新组件 — 管理 N 个虚拟身份，每个身份 = 一致的指纹档案 + 绑定的代理 IP + Cookie 状态 + 行为历史。身份之间完全隔离

4. Proxy Router 从 MediaCrawler 移植并增强 — 加入 TLS 指纹感知，同一身份始终走同一出口 IP（直到过期）

5. 自适应限速 — 不是固定 interval，而是根据平台响应信号动态调整：
- 返回 429 → 指数退避
- 出现验证码 → 降速 50% + 触发 Challenge Solver
- 连续成功 → 缓慢提速（但不超过人类速率上限）

## 实施优先级

按风控对抗价值排序：

1. L2 身份档案一致性 + L4 签名统一走 Pinchtab — 这是最大的短板，修复后立刻提升所有平台的通过率
2. L1 代理池移植到 Go 侧 — 让 Pinchtab 原生支持代理轮换
3. L3 行为模拟增强 — 滚动、停留、闲置轨迹
4. L5 验证码扩展 — 点选文字、旋转图片
5. L1 TLS 指纹 — 对 API 直连场景（非浏览器）最关键
