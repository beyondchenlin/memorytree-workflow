Based on my comprehensive second-pass review of the SQLite migration, here are the findings:

## CRITICAL ISSUES

1. Type Safety: Boolean stored as TEXT without conversion on retrieval
- File: packages/core/src/transcript/db.ts, lines 294-299
- Severity: HIGH
- Issue: repo_mirror_enabled is a boolean in ManifestEntry (types/src/index.ts:50) but stored as TEXT in SQLite. On write (line 298), String(value) converts true → "true" and false → "false". However, on read, there's no reverse conversion. When findLatestSession retrieves rows (recall.ts:148), it casts to Record<string, unknown> and the string "false" is truthy in JavaScript.
- Impact: Any code checking if (row['repo_mirror_enabled']) will treat "false" as truthy, causing logic errors.
- Example: In recall.ts line 70, String(session['global_clean_path'] ?? '') works fine, but if code later does if (manifest.repo_mirror_enabled) after reading from DB, it will be wrong.

2. Edge case: buildCommitLayer returns empty string on non-git directories
- File: packages/core/src/utils/exec.ts, lines 44-71
- Severity: MEDIUM
- Issue: When parsed.cwd is not a git repo, buildCommitLayer silently returns '' (line 69). This is intentional per the design, but there's no logging or indication that commit data wasn't collected. For Phase 1, this is acceptable since commit_layer is optional, but it's a silent failure mode.
- Impact: Users won't know if commit data was skipped due to missing git repo vs. the feature not being implemented yet.

3. Missing test coverage for new columns
- File: packages/core/tests/transcript/db.test.ts
- Severity: MEDIUM
- Issue: Tests verify cleaning_mode and repo_mirror_enabled columns exist (line 112-113), but there are NO tests that:
- Insert a manifest with repo_mirror_enabled: true, then retrieve it and verify it's still true (not the string "true")
- Insert with repo_mirror_enabled: false and verify it's false (not the string "false")
- Test the searchRecall layer fallback when commit_layer is empty
- Impact: The boolean conversion bug won't be caught by tests.

4. Config system not wired to storage factory
- File: packages/sync/src/config.ts defines vector_backend, graph_backend, embedding_model, etc., but these are NEVER passed to createVectorAdapter or createGraphAdapter
- Severity: MEDIUM (for Phase 1)
- Issue: In packages/sync/src/heartbeat.ts, the config is loaded (line 67) but never used to configure storage. The factory functions in packages/core/src/storage/factory.ts accept StorageConfig but heartbeat never calls them. This means config fields like vector_backend: 'qdrant' are loaded but ignored.
- Impact: Phase 1 is unaffected since vector/graph adapters throw "not implemented". But this is a wiring gap that will cause issues in Phase 2.

5. Concurrency: Multiple upsertSearchIndex calls can race
- File: packages/core/src/transcript/db.ts, lines 281-306
- Severity: LOW (mitigated by busy_timeout)
- Issue: Each upsertSearchIndex call opens a new DB connection, runs schema migration, and rebuilds FTS. If two processes call this simultaneously:
- Both open DB with busy_timeout = 5000
- Both try to run ALTER TABLE ADD COLUMN (idempotent, so safe)
- Both rebuild FTS index (line 302) — this is a full rebuild, not incremental
- The second rebuild will overwrite the first's work, but since it's a rebuild from the same table state, it's safe
- Mitigation: WAL mode + 5s busy_timeout prevents corruption, but FTS rebuilds are inefficient under concurrent load.
- Impact: Performance degradation under high concurrency, not data corruption.
