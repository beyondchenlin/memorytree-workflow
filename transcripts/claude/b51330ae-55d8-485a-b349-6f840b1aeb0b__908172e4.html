## CI & Publish Configuration Review — openmnemo Monorepo

---

### CRITICAL

C-1: CLI binary ships stale branding — openmnemo binary runs memorytree program
packages/cli/src/main.ts:11-13

The published binary is named openmnemo (via bin in package.json), but the Commander program is hardcoded as memorytree with description "MemoryTree — transcript import, dedup, indexing, and session continuity". Every user who runs openmnemo --help sees "memorytree", every error message says "memorytree", and the hardcoded version string on line 13 is 0.1.0 while the package version is 0.2.0. The version shown by openmnemo --version will be wrong.

Fix: change .name('memorytree') to .name('openmnemo'), update the description, and replace the hardcoded .version('0.1.0') with a dynamic read from the package's own version field (e.g., via createRequire or import of package.json).

---

C-2: Publish workflow runs no tests — broken code can be released to npm
.github/workflows/publish.yml:29-36

The publish job does pnpm install → pnpm build → pnpm publish. There is no pnpm test or pnpm typecheck step. A tag push triggered after a breaking commit (even one that CI caught on a branch but was force-pushed) will publish a broken package.

Fix: add - run: pnpm test and - run: pnpm typecheck steps between build and publish. Alternatively require CI to pass on the tagged commit via a needs: dependency on a separate ci job, or use a GitHub environment with a required status check.

---

C-3: No tag-to-version consistency check — tag v0.3.0 can publish packages at 0.2.0
.github/workflows/publish.yml (entire file)

The tag pattern v* triggers the publish workflow, but nothing verifies that the tag matches the version in package.json. If someone forgets to bump the version before tagging, npm will reject the publish (version already exists). Conversely, if packages are already bumped to 0.3.0 but the tag is v0.2.5, pnpm will silently publish 0.3.0 under the tag trigger — or fail with a conflict.

Fix: add a pre-publish validation step:

TAG_VERSION="${GITHUB_REF_NAME#v}"
PKG_VERSION=$(node -p "require('./packages/cli/package.json').version")
if [ "$TAG_VERSION" != "$PKG_VERSION" ]; then echo "Tag/version mismatch"; exit 1; fi

### HIGH

H-1: workspace:* resolves to an exact version on publish, not ^version
packages/core/package.json:31, packages/report/package.json:30, packages/sync/package.json:30-32, packages/cli/package.json:33-36

pnpm replaces workspace:* with the exact version at publish time (e.g., "0.2.0", not "^0.2.0"). This means a consumer who installs @openmnemo/cli@0.2.0 will have a peer lock to @openmnemo/core@0.2.0 exactly. If a patch @openmnemo/core@0.2.1 is released independently, npm will not automatically upgrade it; consumers must manually bump @openmnemo/cli. For a monorepo that always releases all packages in lockstep this is acceptable, but it should be a conscious decision.

Fix: use workspace:^ instead of workspace:* for inter-package dependencies, or document the "lockstep release only" policy explicitly.

---

H-2: @types/node@^25.5.0 used in all packages while engines targets Node >=20
packages/core/package.json:36, packages/report/package.json:36, packages/sync/package.json:36, packages/cli/package.json:36

Node 25 type definitions expose APIs that do not exist in Node 20 or 22 (e.g., newer fs, crypto, stream overloads). TypeScript will not warn if code uses a Node 25-only API when @types/node@25 is the source of truth. The CI matrix tests on Node 20 and 22, but the type checker is satisfied by Node 25 types — so type checking passes even if the code would crash at runtime on Node 20.

Fix: align @types/node to the minimum supported version: "@types/node": "^20.0.0".

---

H-3: turbo test is cached by default — stale test results can silently mask failures
turbo.json:8-10

Turbo caches all tasks by default. Since the test task has no cache: false and no explicit outputs, Turbo will consider tests "passed" from cache when inputs haven't changed. This is normally fine, but in the publish workflow (which builds fresh) the absence of a test run combined with turbo cache means tests may not re-execute if the dist is bit-for-bit identical to a previous cached run. More importantly in CI, if a non-deterministic test passes on the first run it will be eternally "green" from cache even after unrelated dependency changes.

Fix: add "cache": false to the test task in turbo.json, or add "outputs": [] with inputs narrowed to source files only.

---

H-4: All GitHub Actions are pinned to mutable version tags, not SHAs
.github/workflows/ci.yml:21-27, .github/workflows/publish.yml:19-27

actions/checkout@v4, pnpm/action-setup@v4, and actions/setup-node@v4 are all referenced by mutable tags. A compromised or mis-tagged release of any of these actions would silently affect all CI runs, including the publish job that writes to npm with id-token: write permission.

Fix: pin each action to a specific commit SHA. For example:

uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af68  # v4.2.2

### MEDIUM

M-1: No engines field in any published package
packages/types/package.json, packages/core/package.json, packages/report/package.json, packages/sync/package.json, packages/cli/package.json (all missing engines)

The root package.json declares "engines": { "node": ">=20" } but none of the published packages inherit or repeat this. npm will display no engine constraint to consumers, who may install on Node 16/18 and get cryptic runtime errors. The engines field in the root of a monorepo does not propagate to published packages.

Fix: add "engines": { "node": ">=20" } to every package's package.json.

---

M-2: No exports field in any package — dual-mode resolution is broken for bundlers
All five packages/*/package.json files

Modern bundlers (Webpack 5, Vite, esbuild), Jest/Vitest with moduleNameMapper, and Node's native ESM loader all prefer the exports field over main/types. Without it, deep-import subpath access is unrestricted (any file in dist/ is importable), and TypeScript 4.7+ with moduleResolution: bundler or nodenext will fail to resolve types correctly in some tool chains.

Fix: add a minimal exports block to each library package:

"exports": {
  ".": {
    "import": "./dist/index.js",
    "types": "./dist/index.d.ts"
  }
}

For @openmnemo/cli (binary-only), exports can be omitted or set to ".": "./dist/main.js" if programmatic import is ever intended.

---

M-3: @openmnemo/report main/types paths lack the leading ./ unlike all other packages
packages/report/package.json:7-8

All other packages use "./dist/index.js" (POSIX relative). @openmnemo/report uses "dist/index.js" (bare relative). Both forms are valid per the npm spec, but the inconsistency could confuse tooling that parses these paths and applies path.resolve logic differently across platforms.

Fix: change to "./dist/index.js" and "./dist/index.d.ts" to match the rest of the monorepo.

---

M-4: tsconfig.json strictness is inconsistent across packages
packages/report/tsconfig.json:7-8 vs all others

@openmnemo/report has noUncheckedIndexedAccess: true and exactOptionalPropertyTypes: true. No other package does. Code in @openmnemo/core, @openmnemo/sync, and @openmnemo/cli that consumes @openmnemo/report's types will silently skip these extra checks, meaning type bugs caught in report could be introduced by callers in other packages.

Fix: add noUncheckedIndexedAccess: true and exactOptionalPropertyTypes: true to all package tsconfig.json files, or extract a shared tsconfig.base.json at the root.

---

M-5: @openmnemo/cli tsup config produces no .d.ts files
packages/cli/tsup.config.ts:3-9

The CLI tsup config has no dts: true. While @openmnemo/cli is primarily a binary, its package.json has no private: true and is published. Any consumer who attempts to import from @openmnemo/cli programmatically will get no TypeScript types. Combined with the absence of a main field, programmatic import is entirely unguarded.

Fix: if @openmnemo/cli is binary-only, mark it "private": false (already) but document that no programmatic API is guaranteed. If programmatic use is intended, add dts: true to tsup.config.ts and add main/types/exports fields.

M-6: Publish workflow does not verify NPM_TOKEN secret is set
.github/workflows/publish.yml:36

If secrets.NPM_TOKEN is not configured on the repository, the publish step will fail with a cryptic auth error after the build has already completed. There is no early validation that the secret exists.

Fix: add an early check step:

- name: Verify NPM_TOKEN
  run: |
    if [ -z "$NODE_AUTH_TOKEN" ]; then
      echo "NPM_TOKEN secret is not set"; exit 1
    fi
  env:
    NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}

M-7: No Dependabot configuration for GitHub Actions or npm dependencies
.github/ (missing dependabot.yml)

There is no Dependabot or Renovate config. Action versions (@v4 tags) and package versions will silently go stale. Given that actions are pinned to mutable tags (H-4), this compounds the risk.

Fix: add .github/dependabot.yml with both package-ecosystem: npm for the workspace and package-ecosystem: github-actions for the workflows.

### LOW

L-1: turbo.json has no globalEnv for NODE_ENV or similar — cache may be non-deterministic
turbo.json (entire file)

If any package reads process.env.NODE_ENV at build time (e.g., for conditional dev/prod output), Turbo will not factor that into the cache key. A CI build with NODE_ENV=test and a local build with NODE_ENV=development would share a cache hit.

Fix: add "globalEnv": ["NODE_ENV"] (and any other env vars that affect build output) to turbo.json.

L-2: @openmnemo/types has no test script — turbo test silently no-ops for it
packages/types/package.json:23-27

Turbo's test task runs npm run test per package. @openmnemo/types has no test script, so Turbo skips it with a warning (or silently depending on version). If type-only utility functions are added to types/, there is no test harness available and no reminder to add one.

Fix: add "test": "vitest run" to packages/types/package.json and create a tests/ directory with at least a smoke test.

L-3: CI matrix includes Windows but no cross-platform path handling is verified
.github/workflows/ci.yml:14

The matrix runs on windows-latest. The fast-glob package normalizes separators, but path.join calls in @openmnemo/core and @openmnemo/sync may produce \-separated paths on Windows that are then compared against POSIX patterns. If tests are written to expect / separators this would fail on Windows CI specifically.

This is a latent risk — whether it actually manifests depends on the test implementations. Worth auditing path comparison logic in packages/core/src/ and packages/sync/src/.

L-4: pnpm/action-setup@v4 without explicit version: reads from packageManager field
.github/workflows/ci.yml:23, .github/workflows/publish.yml:21

This is actually correct behavior for pnpm/action-setup@v4, which automatically reads the packageManager field from package.json (pnpm@9.15.0). No issue here, but it is worth noting that if the packageManager field is ever removed or renamed, both workflows will silently fall back to whatever pnpm version the action defaults to.
