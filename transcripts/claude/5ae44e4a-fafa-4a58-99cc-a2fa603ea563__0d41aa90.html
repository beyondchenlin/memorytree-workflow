#### 1.2 _transcript_import.py 中缺失的功能

Phase 1.4 (import.ts) 的 checkbox 中声称覆盖了清洗逻辑，但漏掉了以下几个具体细节：

- write_clean_markdown() 的 YAML frontmatter 格式（title, cwd, branch, raw_upload_permission 等字段的 yaml_escape 处理）
- preserve_existing_import_timestamp() 逻辑——当 manifest 内容未变时保留原始 imported_at，这是幂等性的关键实现，而文档只写了"幂等性：已导入的 session_id 跳过"，实际 Python 代码是基于 sha256 比对实现的，不是基于 session_id
- manifest_changed() 辅助函数
- append_to_event_log 条件判断（仅当 manifest 实际变化时才追加 sessions.jsonl，防止重复写入）

#### 1.3 _transcript_common.py 中缺失的函数

Phase 1.2 的 checkbox 没有列出这些具体函数：
- earliest_timestamp() / parse_iso_timestamp()（时间戳比较逻辑）
- timestamp_partition()（生成 year_token/month_token 目录结构）
- yaml_escape()（写 markdown frontmatter 用）
- summarize_value() / truncate()（工具事件摘要）
- join_paragraphs() / ensure_dict() / ensure_list() / get_nested()
- extract_text_blocks() / extract_simple_text() / extract_gemini_text() / extract_gemini_parts()（文本提取是最复杂的逻辑）
- find_first_mapping_with_keys()
- load_jsonl() / load_json()
- sha256_file()（注意：这是对整个文件的 sha256，与内容 hash 的 _content_hash() 不同）

这两个 sha256 函数具有不同语义，文档中把它们笼统写成一条会导致实现时混淆。

#### 1.4 _transcript_discover.py 中缺失的函数

Phase 1.6 只列了 discoverTranscripts() 和 matchProject()，漏掉：
- infer_project_slug() — 从 parsed.cwd 或 parsed.source_path 推断项目名，claude 的 re.sub(r"^[a-z]--", ...) 处理逻辑是独特的
- project_slugs_match() — 防止 "unknown-project" 相互匹配的保护逻辑
- safe_file_mtime() — 文件排序用

#### 1.5 _transcript_index.py 的 SQLite schema 细节

Phase 1.5 只写了"SQLite 索引 + FTS5"，但实际 Python 代码用的是普通表 + ON CONFLICT DO UPDATE（UPSERT），没有用 FTS5。主键是 (client, project, session_id, raw_sha256) 四元组。这是重要的 schema 设计决策，迁移文档应明确。

#### 1.6 _transcript_parse.py 中的 infer_client() 函数

Phase 1.3 没有提到 infer_client() 函数（从文件路径自动推断客户端类型），这个函数被 import-transcripts.py 直接使用，在统一 CLI 中会需要。

#### 1.7 ParsedTranscript dataclass 字段不完整

Phase 1.1 的类型定义中写了 ParsedTranscript { session_id, client, project, messages, tool_events, started_at, ended_at, raw_path }，但实际 Python 实现的字段是：client, session_id, title, started_at, cwd, branch, messages, tool_events, source_path。

差异：
- 文档中有 project 字段，实际代码中没有（项目信息由 _transcript_discover.infer_project_slug() 在导入时推断）
- 文档中有 ended_at 字段，实际代码中没有
- 实际代码有 title, cwd, branch 字段，文档中漏掉
- 文档中叫 raw_path，实际叫 source_path（类型是 Path，不是 str）

### 2. 与原 TODO.md 的一致性

#### 2.1 P0.5 跨会话上下文恢复——覆盖严重不足

这是最严重的遗漏。recall-session.py 是 P0.5 功能的核心实现脚本（296L），在迁移文档的 Phase 3.4 仅用一行 memorytree recall (← recall-session.py) 带过，完全没有展开迁移细节。

该脚本包含以下需要独立迁移的逻辑：
- _sync_current_project() — 按当前 repo 过滤的增量同步（与全量 discover 不同）
- _find_latest_session() — SQLite 查询（包含 cwd 路径匹配 + 项目 slug 匹配的双重 fallback 逻辑）
- _find_latest_from_jsonl() — SQLite 不存在时的 JSONL fallback
- _cwd_matches() — 路径精确匹配或子目录匹配
- --activation-time 参数——排除当前会话的时间戳过滤逻辑
- 输出格式：JSON 和文本两种格式，以及 clean_content 的内联读取

建议在 Phase 2 或 Phase 3 中独立出一个 2.7 recall.ts 或 3.5 recall.ts 模块，至少有 5-6 个 checkbox。

#### 2.2 P2 大模型职责最小化——迁移计划完全未覆盖

TODO.md 的 P2 要求：
1. 更新 SKILL.md 的 On Activation 段，明确 heartbeat 职责
2. On Activation 时检测定时任务是否已注册，读取 alerts.json 并展示

这两件事均未在迁移文档中出现。P2 是 AI 指令层面的工作（修改 SKILL.md 等参考文档），不涉及代码迁移，但在 Phase 3.5 清理项里也应该显式标注"更新 SKILL.md 引用的脚本路径"。

#### 2.3 P3 分发方案需要更新

原 TODO.md 的 P3 方案是 pip install / pyproject.toml。迁移到 TypeScript 后应改为 npm publish / npx，但迁移文档只在 Phase 3.5 清理里一笔带过"更新 README.md"，没有对应的具体 P3 分发方案 checkbox。

缺失的工作：

### 3. 遗漏的迁移项

#### 3.1 recall-session.py (296L) — 无专属迁移章节

如前所述，这是最大的单点遗漏，建议新增 Phase 2.7 或 Phase 3.0（因为它依赖 Phase 1 的所有 transcript 模块和 Phase 2 不依赖 heartbeat）。

#### 3.2 discover-transcripts.py (194L) 和 import-transcripts.py (117L) — CLI 层逻辑未覆盖

Phase 3.4 只列了 memorytree import 和 memorytree discover 两行，但这两个 Python 脚本包含大量 CLI-层的业务逻辑（非 library 层），包括：
- --scope current-project|all-projects 过滤逻辑
- --raw-upload-permission 三态处理
- --limit 限流
- --client 过滤
- 结构化 JSON 输出格式（imports + skipped 数组）
- summarize_exception() 辅助函数
- mirror_to_repo 条件逻辑

这些逻辑在 1-2 小时内不可能完成，需要拆分。

#### 3.3 detect-memorytree-locale.py (42L) 的 --format json 输出

Phase 3.2 的 locale.ts 只提到"语言检测"，没有提到 CLI wrapper 的 JSON 输出格式（requested, repo_locale, system_locale, effective_locale 四字段），这是外部调用的 API 合同。

#### 3.4 SKILL.md 中的 Python 脚本引用

SKILL.md 的 Resources 段直接列出了这些脚本路径：
- scripts/heartbeat.py
- scripts/memorytree_daemon.py
- scripts/recall-session.py
- scripts/detect-memorytree-locale.py
- scripts/discover-transcripts.py
- scripts/import-transcripts.py
- scripts/upgrade-memorytree.py
- scripts/init-memorytree.py

迁移文档 Phase 3.5 只写了"更新 SKILL.md: 移除 execution-environment.md 相关内容"，但实际上所有 scripts/*.py 的路径都需要更新为 TS 的等效路径（或者改为命令名）。execution-environment.md 本身也需要完整重写（从"如何找 uv/python 运行时"变成"如何找 Node.js 运行时"）。

#### 3.5 references/execution-environment.md 需要完整重写，不是"移除"

该文件当前内容是关于如何运行 Python 脚本的完整策略文档（4 个场景，3 种 fallback）。迁移到 TypeScript 后，它的核心逻辑变为"检查 node 是否可用 → 直接运行 dist/cli.js"，但 fallback 逻辑也需要重新设计（Node.js 通常比 Python 更普遍，但 WASM 版本的处理方式不同）。Phase 3.5 用一行"移除 execution-environment.md 相关内容"带过是不够的。

#### 3.6 agents/openai.yaml 未被提及

agents/ 目录下有一个 openai.yaml 文件，迁移文档完全没有提到。虽然这不是 Python 代码，但如果它包含对 Python 脚本的引用，需要同步更新。

#### 3.7 assets/templates/ 目录未被提及

_scaffold_utils.py 和 init-memorytree.py 都依赖 assets/templates/<locale>/ 目录中的模板文件（goal.md, todo.md, chat-log.md, agents.md）。Phase 3.1 的 scaffold.ts 只提到了"创建 Memory/ 目录结构，生成模板文件"，但没有提到模板文件是从 assets/templates/ 读取的，TS 版本需要以同样方式处理这些静态资产（可能需要打包策略）。

### 4. 任务粒度评估

#### 过大的任务（需拆分）

Phase 1.3 解析器 — 5 个 checkbox，但 Codex 解析器本身有 7 种 record_type/payload_type 分支（session_meta, response_item/event_msg + message/user_message/agent_message/function_call/custom_tool_call），每个解析器的实现工作量远超 1-2 小时。建议拆成：
- 1.3a Codex 解析器 + 测试
- 1.3b Claude 解析器 + 测试
- 1.3c Gemini 解析器 + 测试

Phase 2.5 心跳主逻辑 — "主函数: runHeartbeat(config)"包含扫描 → 导入 → 检测 → git commit + push 全流程，估计 4-6 小时。建议拆成：
- 2.5a 主流程编排（不含 git）
- 2.5b git 操作（gitCommitAndPush, _try_push, _git）

Phase 3.4 统一 CLI — 7 个子命令 + bin 入口合并为 2 个 checkbox，实际工作量是各命令的 CLI 层 logic（discover/import 的参数透传逻辑、recall 的参数透传）。建议每个子命令一个 checkbox，或按命令族分组。

#### 可以合并的任务

Phase 1.1 类型定义 — 3 个 checkbox 可以合并为 1 个（transcript.ts 类型文件是纯声明，含枚举和接口，估计 30 分钟完成）。

Phase 2.4 日志 — 3 个 checkbox 对于一个仅 64L 的模块过细，合并为 1 个即可。

Phase 0 验证构建流程 — "写一个 smoke test"和"验证构建流程"可以合并为一条。
