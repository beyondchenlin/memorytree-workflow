### CRITICAL

[CRITICAL-1] Doubao started_at assertion will fail: naive string vs UTC-normalized string

Test line 238:

expect(parsed.started_at).toBe('2026-03-16T10:22:17')

In parseDoubaoTranscript (parse.ts line 427), the Created: header value is assigned directly: startedAt = raw where raw = "2026-03-16T10:22:17". This bypasses normalizeTimestamp, so the value is stored verbatim as '2026-03-16T10:22:17' (no Z).

However, the Doubao turn timestamp fields follow the same pattern (parse.ts line 451: const timestamp = (match[2] ?? '').trim().replace(' ', 'T')), so message timestamps will also be naive ISO without Z.

The assertion toBe('2026-03-16T10:22:17') is technically correct for the current parser behavior — the raw value is assigned directly. But the Gemini parser takes the same timestamp through normalizeTimestamp → parseIsoTimestamp → treats it as UTC → outputs '2026-03-16T10:22:17Z'. This inconsistency means the Doubao test assertion is testing the current (potentially unintended) behavior of not running through normalizeTimestamp. If the parser is later corrected to call normalizeTimestamp(raw), this assertion will break with '2026-03-16T10:22:17Z'. The test documents the current behavior but does not flag the inconsistency as something to watch.

Severity: CRITICAL — the assertion is coupled to an implementation detail that differs from every other parser, and it will silently start failing if the parser is normalized.

[CRITICAL-2] Gemini slug inference test is correct in isolation but tests the wrong path for real Gemini files

Test lines 137-138 create the source file at:

<tmpDir>/myproject/conversation.json

inferProjectSlug for Gemini (discover.ts lines 117-122) uses basename(dirname(parsed.source_path)) — the parent directory name. Here that is 'myproject'. Test line 180 asserts slug === 'myproject'. This is correct for this fixture.

However: the test's Gemini fixture also provides cwd: '/home/user/myproject' (fixture line 145). inferProjectSlug checks parsed.cwd first (discover.ts line 102). Since cwd is non-empty, it uses basename('/home/user/myproject') = 'myproject'. The parent-directory slug path for Gemini is never actually exercised by this test — the cwd path is hit instead.

The comment "Infer project slug from parent directory name" (test line 178) is therefore misleading: it claims to test the Gemini-specific parent-directory logic, but the cwd branch wins. A separate fixture without a cwd field would be needed to actually test that path.

Severity: CRITICAL — the test comment documents intent that is never executed, creating false confidence in the Gemini-specific slug branch.

Doubao test line 258:

const cleanContent = readFileSync(manifest.global_clean_path, 'utf-8')

Looking at import.ts lines 80 and 78:

- repo_clean_path is assigned toPosixPath(relative(root, repoCleanPath)) — a relative path (relative to root, which is the repoRoot parameter of importTranscript).

In the pre-existing Codex and Gemini tests, repo_clean_path is read as (e.g. test line 68):

This is correct. The Doubao test avoids repo_clean_path entirely (mirror is disabled: mirrorToRepo=false), so this distinction does not cause a test failure for Doubao. However, the inconsistency between global_clean_path (absolute) and repo_clean_path (relative) is not surfaced by any comment or assertion in the new tests, increasing maintenance risk.

Severity: HIGH — the path semantics are not documented by the tests, and a future developer could incorrectly use global_clean_path relative to globalRoot or repo_clean_path as absolute, causing silent test failures.

[HIGH-2] Gemini tool event double-counting: the toolUse node will be visited twice

The Gemini fixture (test lines 155-158) places toolUse inside a history entry that already has a role:

{
  "role": "model",
  "parts": [{ "text": "..." }],
  "toolUse": { "name": "read_file", "input": { "path": "test.ts" } }
}

In parseGeminiTranscript, the visit function (parse.ts line 310) processes a node and then recursively visits all its values (line 350: for (const value of Object.values(rec)) { visit(value) }).

For the model node:
1. toolName is found via getNested(rec, 'toolUse', 'name') → pushes one tool event with "read_file".
2. Then recursive visit is called on each value, including toolUse: { name: 'read_file', input: {...} }.
3. Inside that nested visit, rec['toolName'] is undefined but rec['tool'] is also undefined; however rec['name'] is 'read_file'. The check is: getNested(rec, 'toolUse', 'name') — this resolves to null (no toolUse inside toolUse). Then rec['toolName'] and rec['functionName'] are checked — both undefined. rec['tool'] is undefined.

Actually on re-reading: the inner object { name: 'read_file', input: {...} } has no role and no toolName/functionName/tool field. getNested on this inner node for 'toolUse' returns null. So no second tool event is pushed from the inner node.

But wait — let me check the input object { path: 'test.ts' } visited next. It has no role and no tool indicator fields. So no double-counting occurs for this specific fixture.

The test assertion toBeGreaterThanOrEqual(1) (line 174) would pass regardless of double-counting, making it a weak guard. The assertion does not verify the exact count is 1, so if the parser does produce 2 tool events via a future fixture change, this test would still pass. This is a looseness issue rather than a current bug.

Severity: HIGH — using toBeGreaterThanOrEqual(1) instead of toBe(1) for tool events leaves double-counting bugs undetectable.

[HIGH-3] Doubao session_id extraction: URL query string not handled

The parser extracts session_id via url.split('/').pop() (parse.ts line 417). The test URL https://www.doubao.com/chat/38416801786598914 has no query string, so pop() returns '38416801786598914' correctly.

But real Doubao URLs may include query parameters such as ?source=... or fragment identifiers. For example:

https://www.doubao.com/chat/38416801786598914?conv=abc

In that case pop() returns '38416801786598914?conv=abc', making session_id include the query string. The test does not cover this case. If this is a real-world scenario, both the parser and a corresponding test are missing it.

Severity: HIGH — if real exports include query strings, session IDs will be incorrect and SQLite lookups will silently fail deduplication.
