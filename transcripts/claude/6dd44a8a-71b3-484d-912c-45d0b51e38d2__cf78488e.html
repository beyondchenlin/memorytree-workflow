## Comprehensive Report: MemoryTree Report Module Structure

Based on my thorough exploration of the memorytree-workflow skill, here's the complete analysis for porting the report module:

### 1. src/types/report.ts - Complete Content

BuildReportOptions interface:
- root: string - Project root directory
- output: string - Output directory for generated report
- noAi?: boolean - Skip AI operations (default: false)
- model?: string - Claude model ID (default: claude-haiku-4-5-20251001)
- locale?: string - Locale code (default: 'en', supports 'zh-CN')
- ghPagesBranch?: string - GitHub Pages branch for deployment (empty = skip)
- cname?: string - Custom domain for CNAME file (empty = skip)
- webhookUrl?: string - Webhook endpoint for notifications (empty = skip)
- newSessionIds?: string[] - Session IDs newly imported (for webhook context)
- reportBaseUrl?: string - Base URL for RSS/OG links (empty = skip)

Additional types:
- ReportStats - Aggregate metrics (sessions, messages, tool events, active days, client counts, day/week buckets)
- SummaryCache - SHA256-based AI summary cache with timestamp
- LinkGraph - Backlinks/forward links maps between sessions
- SearchIndexEntry - Full-text search index entries

### 2. src/cli/cmd-report.ts - Complete Content

CmdReportBuildOptions (CLI → build.ts adapter):
- Converts CLI args to BuildReportOptions
- Resolves paths to absolute paths
- Spreads optional fields with conditional spreading

Flow:

cmdReportBuild(opts) → buildReport({
  root, output, noAi, model,
  ...locale spread,
  ...reportBaseUrl spread
})

Design note: CLI doesn't expose ghPagesBranch, cname, or webhookUrl - those come from config.ts

### 3. src/heartbeat/config.ts - New Config Fields (v2)

Config interface adds 4 new fields:
- locale: string (default: 'en')
- gh_pages_branch: string (default: '')
- cname: string (default: '')
- webhook_url: string (default: '')
- report_base_url: string (default: '')

These are persisted in ~/.memorytree/config.toml and loaded before report building.

### 4. src/types/transcript.ts - Client Type

export type Client = 'codex' | 'claude' | 'gemini' | 'doubao'

Yes, 'doubao' is included. (Doubao is Alibaba's AI chat model.)

Main entry point: async buildReport(options: BuildReportOptions): Promise<void>

Key imports & dependencies:

Data flow:

1. Load manifests from Memory/06_transcripts/manifests/*.json (recursive)
2. Parse clean markdown from each manifest (.../clean/{client}/{session}.md)
3. Extract tool events from raw JSONL (if available) for stats
4. Parallel summaries + tag extraction with semaphore (max 5 concurrent API calls)
5. Render all pages (dashboard, transcripts, goals, todos, knowledge, archive, projects, graph, search, RSS)
6. Deploy to GitHub Pages (if branch specified)
7. Send webhook notifications (if URL specified)

Does NOT depend on:
- Any heartbeat/sync modules
- Any transcript parsing/import logic (just reads already-processed artifacts)

### 13. Critical Observations for Porting

1. Zero external rendering libs: All HTML is hand-crafted. SVG charts drawn in JavaScript.
2. Graceful degradation: Missing AI key → skip summaries/tags. Missing SDK → empty summaries. CLI doesn't pass deploy config (comes from heartbeat/config).
3. Path handling: Uses both repo_clean_path (relative) and global_clean_path (absolute).
4. Immutable patterns: Link graph and stats accumulation use spread operators (not mutation).
5. Cache location: Lives outside output dir so rm -rf Memory/07_reports doesn't wipe caches.
6. Security: Webhook URL validation (no private IPs, https only), git branch name whitelist.
7. ESM-only: .js extensions required on all imports; no CommonJS.
8. Concurrency: Semaphore on AI calls (max 5), git operations sequential in deploy.
