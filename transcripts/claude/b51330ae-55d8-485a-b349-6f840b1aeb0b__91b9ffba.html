## Review Findings

HTTP

CRITICAL: Servers on ports 19871 and 19872 are never closed

cmdReportServe returns number (synchronously), not an http.Server. The test-level server variable declared in the describe block is never assigned by these tests — cmdReportServe gives back 0, not a server handle. The afterEach cleanup checks if (server), but server is always null, so the guard never fires.

Concrete effect:
- The test on port 19871 ("serves an HTML file") calls cmdReportServe and deliberately punts with resolve() inside the "close the server" promise at line 192-193. The comment openly admits it cannot close the server.
- The test on port 19872 ("returns 404") likewise never closes its server.
- Both servers keep their ports bound for the lifetime of the Node process. If the test suite is re-run immediately (or in watch mode) or another test file runs in the same worker and happens to bind these exact ports, the EADDRINUSE path in cmd-report.ts calls process.exit(1) — which kills the whole test worker.

The "returns 0 synchronously" test (port 0) also starts a server that is never closed, though port 0 makes a conflict unlikely.

HIGH: cmdReportServe calls process.exit(1) on EADDRINUSE

If port 19871 or 19872 is already in use when the test runs (CI agent reuse, developer running two suites, etc.), the server emits an error and the production code calls process.exit(1). This kills the whole Vitest worker rather than surfacing a test failure, meaning other unrelated tests in the same run are also lost.

MEDIUM: Hardcoded ports 19871/19872 are collision-prone across CI matrices

The ports are not drawn from a system-assigned ephemeral range. On a shared CI host running multiple test suites in parallel (e.g., Node matrix) both ports can be occupied.

Recommended fix: Expose cmdReportServe as returning the http.Server instance alongside the status code (or as a second out-parameter), bind port 0 in tests to get a random free port, capture the actual bound port from server.address(), and explicitly close the server in afterEach.

### 2. vi.doMock Pattern in discover / recall / search

HIGH: vi.doMock called AFTER resetModules but BEFORE re-import — works for isolated tests but breaks in the first test of discover.test.ts

discover.test.ts test 1 ("returns 1 when root does not exist") uses the top-level static import of cmdDiscover at line 11, not a fresh dynamic import. vi.resetModules() is called in beforeEach, but the first test never re-imports — it uses the already-module-cached instance. For this particular test the mock is irrelevant (it returns early before calling core), so it passes today. However if the implementation ever calls a core function before the existence check, the static import would bypass all mocks entirely.

HIGH: vi.doMock in discover.test.ts tests 2 and 3 — resetModules in beforeEach clears the previous mock but vi.doMock for @openmnemo/core is called without first clearing it

vi.resetModules() clears the module registry but does NOT clear pending vi.doMock registrations. The second test registers a mock, then the afterEach calls vi.resetModules() again but not vi.clearAllMocks() or vi.unmock(). The third test then re-registers a new mock. Because both tests call doMock with identical keys, the second registration replaces the first, which is fine here — but this pattern means a mock leak across tests is possible if the afterEach vi.resetModules() timing is imprecise.

MEDIUM: recall.test.ts — test 1 does a fresh dynamic import BEFORE setting up any mock

// beforeEach calls vi.resetModules()
const { cmdRecall } = await import('../../src/cmd-recall.js')

This import happens with no vi.doMock in place. The real @openmnemo/core is loaded. If the real recall function tries to open a database or file in the empty tmpDir, it would throw. In the specific case here the command returns early (nonexistent root), so the real recall is never invoked — but a future refactor that hoists the existence check could silently start calling production code.

MEDIUM: Module identity after resetModules across test boundaries

The vitest config has no pool: 'forks' or isolate: true override. By default vitest runs tests in a single worker thread per file with module isolation only within a file. With vi.resetModules() called in both beforeEach and afterEach, there is a window where the module registry is freshly reset before vi.doMock fires. If vitest ever parallelises within a file (planned feature), this could cause races.

### 4. captureOutput Helper — Safety and Correctness

HIGH: Not concurrency-safe — process.stdout.write / process.stderr.write are global process state

captureOutput replaces process.stdout.write and process.stderr.write directly. Vitest by default runs test files in parallel workers (separate processes), so between files this is fine. However, within a single file all tests run sequentially, and the finally { cap.restore() } pattern is correct for sequential use.

The real hazard is: if a test's async code continues executing after cap.restore() (e.g., an un-awaited promise, a setTimeout, or a background stream callback), it would write to the already-restored handle. The cmdReportServe server's listen callback (console.log('Serving...')) fires asynchronously after the 100ms timeout and after cap.restore() has been called. In the "serves an HTML file" test, cap.restore() is called in the finally block around the synchronous cmdReportServe call, but the server's listen event fires roughly 100ms later. At that point the captures are already restored — so the console.log in the server goes to the real stdout, not the captured buffer. This is a silent inconsistency: the assertion cap.out().toContain('Report generated at:') in cmdReportBuild passes cleanly, but for cmdReportServe the listen-callback output is not captured even though tests might implicitly depend on it.

MEDIUM: captureOutput is duplicated verbatim across all 5 files

Each file defines its own local copy. The report.test.ts version correctly also intercepts console.log/console.error, but the other four files only intercept the raw process.stdout.write/process.stderr.write. If a command is ever changed to use console.log instead of process.stdout.write, all tests except report tests would silently miss the output. The helper should live in a shared test utility file.

LOW: captureOutput accepts Buffer and Uint8Array from Node internals but casts to string

The type cast (c: string) => is a TypeScript lie: process.stdout.write also accepts Uint8Array. If any production code ever calls process.stdout.write(Buffer.from(...)), the outChunks.push(c) call would push a Buffer object, and outChunks.join('') would produce [object Object][object Object]. The signature should accept string | Uint8Array and handle conversion.

The current vitest.config.ts has no pool setting, defaulting to pool: 'threads' (worker threads, one per file). Files are isolated. But ports 19871/19872 are OS-level and exist outside the worker boundary — two test files in different workers sharing these ports would race.
