## MemoryTree Workflow — 功能全景

### 一句话定义

为任意代码仓库添加 持久化、Git 追踪的项目记忆层，让 AI 编程助手（Claude Code、Codex、Gemini CLI）跨会话保留上下文。

---

### 核心架构


你的代码仓库/
├── AGENTS.md                    ← AI 行为规则（读取顺序、记忆规则、Git 规则）
└── Memory/
    ├── 01_goals/                ← 版本化的项目目标
    ├── 02_todos/                ← 与目标绑定的任务列表
    ├── 03_chat_logs/            ← 只追加的会话日志
    ├── 04_knowledge/            ← (可选) 持久化知识笔记、架构决策
    ├── 05_archive/              ← (可选) 已完成/退役的目标和待办
    └── 06_transcripts/          ← (可选) AI 对话历史归档
        ├── clean/               ←   清洗后的 Markdown 索引
        ├── manifests/           ←   导入元数据
        └── raw/                 ←   原始 transcript 镜像

全局数据存放在 ~/.memorytree/（跨项目共享）：


~/.memorytree/
├── config.toml              ← 全局配置
├── alerts.json              ← 待处理通知
├── heartbeat.lock           ← 心跳单实例锁
├── logs/                    ← 心跳执行日志
└── transcripts/             ← 全局 transcript 归档
    ├── raw/                 ←   按 client/project/年/月 存放
    ├── clean/               ←   清洗后的 Markdown
    └── index/
        ├── sessions.jsonl   ←   会话元数据
        └── search.sqlite    ←   全文搜索索引

---

### 功能模块详解

#### 1. 自动检测与初始化

激活 Skill 时（运行 /memorytree-workflow），自动执行：

| 步骤 | 动作 |
|------|------|
| 检测状态 | 判断仓库是 not-installed、partial 还是 installed |
| 检查告警 | 读取 ~/.memorytree/alerts.json，有通知则展示 |
| 未安装 | 运行 init-memorytree.py 脚手架，创建 AGENTS.md + Memory/ |
| 部分安装 | 运行 upgrade-memorytree.py，补齐缺失文件，不覆盖已有策略 |
| 已安装 | 读取当前目标、待办、最新日志，汇报状态，询问下一步 |

不需要人工确认就直接行动——检测到状态后立即处理。

#### 2. 项目记忆管理（日常维护）

| 记忆类型 | 行为规则 |
|----------|----------|
| Goals（目标） | 只在用户明确确认需求/范围变化后才修改；版本化管理 |
| Todos（待办） | 与当前目标保持同步；进度变化时更新 |
| Chat Logs（对话日志） | 只追加，永不重写或删除历史记录 |
| Knowledge（知识） | 存放架构决策、产品想法、运维约束 |
| Archive（归档） | 过期内容从活跃文件移入归档，保持活跃文件精简 |

#### 3. 多客户端 Transcript 归档

支持从三个 AI 客户端导入对话历史：

| 客户端 | 数据源 |
|--------|--------|
| Codex | ~/.codex/sessions/*/rollout-.jsonl |
| Claude Code | ~/.claude/projects/<project>/*.jsonl |
| Gemini CLI | ~/.gemini/ 下的 saved chat 或 checkpoint |

关键原则：
- 原始 transcript 作为证据不可篡改保留
- 清洗后的 Markdown 由确定性代码生成（不消耗模型 token）
- 当前仓库只保留属于本项目的 transcript；其他项目的只进全局归档
- 首次导入时询问用户：是否允许把原始 transcript 提交到仓库

#### 4. 后台心跳进程（合约规范，代码未实现）

heartbeat.py + memorytree-daemon CLI 负责无人值守的自动化：

执行流程：


获取锁 → 加载 config.toml → 遍历注册项目 → 发现新 transcript
→ 导入 → 清洗 → commit → (auto_push 时) push → 释放锁 → 退出

Daemon CLI：

| 命令 | 说明 |
|------|------|
| memorytree-daemon install | 注册到 OS 调度器（cron / launchd / Task Scheduler） |
| memorytree-daemon uninstall | 移除定时任务 |
| memorytree-daemon run-once | 立即执行一次心跳 |
| memorytree-daemon watch | 连续循环（仅开发调试用） |
| memorytree-daemon status | 查看注册状态和上次执行结果 |

设计约束：
- 单次执行、无状态、幂等——安全重跑
- 一台电脑只能注册一个心跳任务
- 不会静默注册，安装前必须确认

#### 5. 敏感信息扫描

心跳清洗 transcript 时自动扫描 API key / password / token 等模式：
- 发现匹配 → 记录警告 + 写入 alerts.json
- 不自动删除或脱敏——由用户决定处理方式

#### 6. Git 安全策略

MemoryTree 的 Git 操作遵循严格的隔离原则：

| 规则 | 说明 |
|------|------|
| 文件隔离 | 只暂存/提交 Memory/** 和 AGENTS.md，不混入产品代码 |
| 分支隔离 | 使用专用分支 memorytree/<scope>-<date>-<slug> |
| 提交格式 | memorytree(<scope>): <subject> 或仓库兼容格式 |
| PR 隔离 | 单独 PR，不混入产品变更 |
| Auto-merge | 仅在仓库保留 review + CI 检查时才启用 |
| Auto-push | 默认开启；无 remote 则跳过并告警；只做 fast-forward |
| 受保护分支 | 绝不覆盖仓库的分支保护和 CI 规则 |

Stop And Ask（8 种场景会暂停询问用户）：混入产品代码、AGENTS.md 有冲突、目标分支不明、仓库禁止自动 PR、CI 冲突、raw transcript 权限未知、跨项目 transcript、受保护分支拒绝直推。

#### 7. 国际化

| 功能 | 说明 |
|------|------|
| 模板语言 | 英文 (en) 和简体中文 (zh-cn) 两套完整模板 |
| 自动检测 | 优先从仓库内容检测语言，其次系统语言 |
| 回复语言 | 跟随用户语言 > 仓库 locale > 系统 locale |
| 不强制翻译 | 不会为了翻译而重写已有文件 |

#### 8. 跨项目搜索

当用户要搜索所有记忆/对话历史时：

1. 先查 search.sqlite 全文索引
2. 用 sessions.jsonl 做会话级元数据过滤
3. 加载匹配的 clean Markdown；精确措辞查 raw
4. 结合当前仓库 Memory/ 文件综合回答
5. 索引不存在时降级扫描 clean/ 目录

#### 9. 执行环境适配

MemoryTree 不假设目标仓库有 Python 环境：


优先 uv run python → 退而求其次 system python → 都没有则从模板直接创建文件

永远不会自动在项目里创建 .venv 或 pyproject.toml。

### 设计哲学

| 原则 | 体现 |

## 几个需要想清楚的设计问题

### 1. 即时同步的执行方式

有两个选择：

- A：Skill 直接调用 discover/import 脚本——即时、可靠，但绕过了心跳的统一入口
- B：调用 memorytree-daemon run-once——复用心跳逻辑，但会同步所有项目，可能比"只同步当前项目"慢

我倾向 A，因为用户要的是快速恢复上下文，不需要全量同步。可以设计一个轻量的"单项目即时同步"路径。

### 2. 总结是否消耗 model token

现有规则说"transcript 清洗由确定性代码完成，不消耗 token"。但这里的总结不是清洗——它是一个有智能的摘要，必须用模型来做。

这和心跳的"清洗"职责不同：
- 清洗：格式转换、去噪、生成索引 → 确定性代码
- 总结：理解对话语义、提取未解决问题 → 必须用模型

所以这个功能本质上是模型在交互会话中读取 transcript 并生成延续摘要，符合现有规则"大模型只做 chat_log 摘要"的精神。

### 3. 和现有 chat_log 的关系

Memory/03_chat_logs/ 已经有会话日志，但：
- 如果会话 A 没有正常结束（用户直接关窗口），chat_log 可能没写完
- chat_log 是会话期间写的摘要，不是事后的全面复盘
- transcript 才是完整的原始记录

所以这个功能的数据源应该是 transcript（优先）+ chat_log（补充）。

### 4. "最近的聊天"的范围

用户说"最近的聊天"可能指：
- 最近一次（最常见）
- 最近几次（"我这两天的讨论"）
- 最近关于某个主题的（"上次讨论部署的那次"）

第一版可以只处理"最近一次、本项目"，后续扩展。

### 5. 触发方式

两种可能：
- 被动触发：用户明确说"看看最近的聊天"
- 主动触发：Skill 激活时如果检测到有未同步的新 transcript，主动提示"发现你上次在这个项目有一个未完成的会话，要看看吗？"
