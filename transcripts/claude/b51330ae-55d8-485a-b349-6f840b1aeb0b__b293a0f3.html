Finding: WARN (minor issue - not critical in current context)

Analysis:
- The lazy rebuild check (lines 185-189) reads transcripts_fts_segdir to detect if the FTS index is empty
- This happens inside searchTranscripts, which loads the database from disk into memory (line 176)
- Race condition scenario: If two processes call searchTranscripts simultaneously on a legacy DB (no FTS table), both will detect segCount === 0 and both will call REBUILD_FTS_SQL
- However, this occurs in-memory only — the rebuild doesn't write to disk, so concurrent reads don't corrupt the file
- Both processes rebuild independently in their own in-memory database, then discard it
- Impact: Minor inefficiency (redundant rebuilds), not a data corruption risk

Why it's safe here:
- upsertSearchIndex (the write path, lines 223-257) creates the FTS table and rebuilds it after EVERY write (lines 249-250)
- This ensures any DB written to disk already has a properly populated FTS index
- The lazy rebuild in searchTranscripts is a fallback for legacy DBs only, not the normal path

Recommendation: Add a comment explaining why in-memory rebuilds are thread-safe.
