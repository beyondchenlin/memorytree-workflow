### HIGH Issues

#### H1 — markdown.ts: Inline code backticks bypass escHtml after bold/italic substitution

File: src/report/render/markdown.ts, lines 211-216

The applyInline pipeline works like this: first escHtml(part) is called on the non-link text, then bold/italic/code regexes run on the already-escaped string. This means * and and ` ` delimiters in user content that appear inside entity-encoded text are still substituted. The substitutions themselves are safe (<strong>, <em>, <code>) but the content captured by (.+?) in each regex is the entity-encoded string — safe. However, there is one subtle ordering problem: the backtick replacement captures ([^]+) and emits <code>$1</code> where $1 is already-escaped content. That part is fine.

The real risk is that applyInline is called on raw heading text (line 100: applyInline(text) where text comes from headingMatch[2]), and text is _not_ pre-escaped before entering applyInline. Inside applyInline, links are extracted first (before escaping). A heading like ## [evil](javascript:alert(1)) would produce <a href="javascript:alert(1)" ...>. The link regex (\[([^\]]+)\]\(([^)]+)\)) does not validate the href scheme — only escHtml is called on it, not a scheme allowlist.

Impact: A user whose Memory files contain markdown headings (goals/knowledge/archive/todos) with javascript: hrefs would get an XSS vector rendered into the static site. This is a local-output-only report, so exploitability is limited, but still a correctness issue worth flagging.

Recommendation: Add a scheme check in applyInline before emitting the <a> tag: only allow http:, https:, mailto:, and relative URLs.

#### H2 — markdown.ts / toc.ts: injectHeadingIds regex does not match headings containing inline HTML

File: src/report/render/toc.ts, line 93

The regex /<h([23])>([^<])<\/h[23]>/g only matches headings whose inner content contains no < characters. But markdownToHtml (called before injectHeadingIds) may output headings like <h2><strong>Bold</strong> Title</h2> if bold markup is in the heading. The [^<] pattern fails to match, so no id attribute is injected and the TOC anchor links produce dead #section-one hrefs.

Impact: TOC links silently fail to scroll for any heading that contains bold, italic, code, or link inline formatting.

Recommendation: Change the inner capture group to ([\s\S]?) (non-greedy) or ([^]?), and update the matching logic to strip inline tags from the inner content before comparing with escHtml(e.text).

#### H3 — tags.ts: No validation of individual tag string length or content

File: src/report/tags.ts, lines 83-87

Tags are filtered typeof t === 'string' && t.length > 0 but there is no upper bound on individual tag length. A model could return a 500-character "tag" string. After toLowerCase() and trim(), this string lands in data-tag="${escTag(t)}" and as visible text in the DOM. HTML-escaping is present (escTag), so no XSS. But a very long tag (e.g. 500 chars) would break the layout and inflate the search index unnecessarily.

Recommendation: Add .filter(t => t.length <= 50) or similar after the trim step.

#### M1 — toc.ts: Duplicate ID algorithm is off-by-one for three or more identical headings

File: src/report/render/toc.ts, lines 41-43

const count = seen.get(base) ?? 0
const id = count === 0 ? base : `${base}-${count}`
seen.set(base, count + 1)

For three identical headings ## Intro, the IDs generated are:
- 1st: count=0 → id=intro, stored count=1
- 2nd: count=1 → id=intro-1, stored count=2
- 3rd: count=2 → id=intro-2, stored count=3

This is correct for uniqueness. The tests cover this case and pass (lines 24-26 of toc.test.ts). However, the suffix counter diverges from GitHub-Flavored Markdown's convention which starts at -1 for the first duplicate. The docs comment says -1, -2, … so the intent is intentional, but there is a discrepancy: the first _original_ heading gets no suffix, the first _duplicate_ gets -1, the second duplicate gets -2. This is consistent and correct per the documented intent. No bug, but worth noting the divergence from GFM (which would put -1 on the second occurrence, -2 on the third) — that's exactly what this code does. Actually the code is correct; this is a non-issue on closer inspection.

#### M2 — rss.ts: toRfc2822 silently falls back to new Date() on invalid ISO

File: src/report/render/rss.ts, lines 83-88

function toRfc2822(iso: string): string {
  try {
    return new Date(iso).toUTCString()
  } catch {
    return new Date().toUTCString()
  }
}

new Date('invalid') does not throw — it returns an Invalid Date object. new Date('invalid').toUTCString() returns the string "Invalid Date" in V8, not a valid RFC 2822 date string. This will produce a malformed <pubDate> element that RSS readers may reject for the entire feed if they are strict.

Recommendation: Check isNaN(d.getTime()) after construction and fall back to new Date().toUTCString() in that branch.

#### M4 — layout.ts: htmlShell dual-signature design creates lang parameter precedence ambiguity

File: src/report/render/layout.ts, lines 289-300

export function htmlShell(
  title: string,
  content: string,
  nav: string,
  extraHeadOrOpts: string | HtmlShellOptions = '',
  lang = 'en',
): string {
  const opts: HtmlShellOptions = typeof extraHeadOrOpts === 'string'
    ? { extraHead: extraHeadOrOpts, lang }   // ← uses the positional `lang` param
    : extraHeadOrOpts                         // ← uses opts.lang if present
  const resolvedLang = opts.lang ?? lang

When called as htmlShell(title, content, nav, { extraHead: '...' }) (without lang), opts.lang is undefined, and resolvedLang falls back to the positional lang = 'en' default. That is correct. But when called with the legacy string form htmlShell(title, content, nav, '<script>...</script>', 'zh-CN'), the object created is { extraHead: '...', lang: 'zh-CN' }, so resolvedLang = 'zh-CN'. Also correct. The design works, but it is fragile: a call site could accidentally pass both a HtmlShellOptions.lang and a fifth positional argument and the positional one would be ignored. This is a maintainability concern.

#### M5 — tags.ts: Concurrent writes to the same cache file are not atomic

File: src/report/tags.ts, lines 136-145

writeFileSync is called from multiple concurrent Promise.all invocations in build.ts (line 144-149). Two concurrent sessions with the same sha256 (which would be unusual but possible if manifests share a raw file) could race on writeTagCache. On most OS/Node.js combinations writeFileSync is not atomic — a partial write would leave a truncated JSON file. On the next read, the corrupt file would be caught by the try/catch in readTagCache and return null, causing a redundant API call, which is acceptable. But the corrupt file remains on disk.

Impact: Low in practice (sha256 collisions between manifest entries are uncommon), but the pattern is fragile. This is the same pattern used by summarize.ts's cache, which was accepted in Phase 1.

#### L2 — layout.ts inline JS: document.startViewTransition intercepts all relative <a> clicks, including anchor-only links

File: src/report/render/layout.ts, lines 100-109

if (!href || href.startsWith('#') || href.startsWith('http')) return;

The guard correctly exempts # fragments and http URLs. However it does not exempt https: URLs (only http:). An absolute https:// link in rendered content would be intercepted and navigated via window.location.href = href. The View Transitions API then fires a cross-origin navigation, which is a no-op for the transition but still sets window.location. This is harmless but unexpected.

Recommendation: Change the check to href.startsWith('http') already covers both schemes (http:// and https:// both start with http). On re-reading, 'https://'.startsWith('http') is true. So this is actually fine — no issue.

#### L3 — cmd-report.ts serve command: statSync called after existsSync, TOCTOU race

File: src/cli/cmd-report.ts, lines 85-87

if (existsSync(filePath) && statSync(filePath).isDirectory()) {

Classic TOCTOU: the file could be deleted between existsSync and statSync. The statSync would throw. The stream.on('error', ...) handler below would not catch a sync statSync throw. However the server is a local dev tool serving a static generated directory, so exploitability is nil. Nonetheless it would cause an unhandled exception and crash the server process.

Recommendation: Wrap in a try/catch or combine into a single statSync with try/catch.
