## Code Review: packages/core/tests/transcript/integration.test.ts — New Tests (lines 135–302)

---

### CRITICAL

C1 — Gemini test: inferProjectSlug assertion does not actually exercise the Gemini code path

Line 179–180:

const slug = inferProjectSlug(parsed)
expect(slug).toBe('myproject')

inferProjectSlug falls back to the Gemini path only when parsed.cwd is empty. Because the test fixture has cwd: '/home/user/myproject', the slug is derived from basename(cwd), not from basename(dirname(sourceFile)). This means the Gemini-specific branch in inferProjectSlug (lines 117–123 of discover.ts) is never reached by this test.

Either add a separate sub-case where cwd is absent and the parent-directory name drives the slug, or document that the cwd-based path is intentionally tested. As written, the test comment "Infer project slug from parent directory name" is factually wrong — it tests the cwd path.

---

C2 — Gemini test: toBeGreaterThanOrEqual allows silent data loss

Lines 189–190:

expect(manifest.message_count).toBeGreaterThanOrEqual(2)
expect(manifest.tool_event_count).toBeGreaterThanOrEqual(1)

The fixture has exactly 2 messages and 1 tool event. There is no deduplication or merging reason to expect more. Using toBeGreaterThanOrEqual means the test passes even if the parser emits spurious duplicates. The Gemini visit() traversal is recursive over all object values (see parse.ts line 350), so duplicate emission is a real risk — the toolUse block nested inside a history entry is visited both when the entry itself is processed and potentially again when child values are recursed. The test should pin the exact values and let any failures surface this.

Replace with:

expect(manifest.message_count).toBe(2)
expect(manifest.tool_event_count).toBe(1)

The same issue appears at lines 170 and 174:

expect(parsed.messages.length).toBeGreaterThanOrEqual(2)
expect(parsed.tool_events.length).toBeGreaterThanOrEqual(1)

### HIGH

H1 — Doubao test: global_clean_path is used as an absolute path without joining globalRoot

Line 258:

const cleanContent = readFileSync(manifest.global_clean_path, 'utf-8')

global_clean_path is stored as an absolute path in the manifest (see import.ts line 46). The Gemini test at line 193 correctly does:

readFileSync(join(repoRoot, manifest.repo_clean_path), 'utf-8')

The Doubao test uses a different field (global_clean_path) which happens to be absolute, so this works — but it is inconsistent with every other test in this file, which all read the repo-mirrored path. It also means the test does not verify that the repo mirror path is populated or readable (because mirrorToRepo is false for the Doubao test). This divergence should be documented explicitly or a separate assertion should verify what happens to repo_clean_path when mirrorToRepo is false (it should be an empty string — that is never asserted).

H2 — Doubao CRLF test: no import-level assertions about the clean markdown content

The CRLF test (lines 275–302) only verifies:
- parsed.messages count and text (parse only)
- manifest.client and manifest.message_count

It does not verify that CRLF stripping survived all the way through to the written clean markdown file. The main value-add of a CRLF integration test is confirming the produced .md file does not contain stray \r characters. Add:

const cleanContent = readFileSync(manifest.global_clean_path, 'utf-8')
expect(cleanContent).not.toContain('\r')
expect(cleanContent).toContain('Hello CRLF world')
expect(cleanContent).toContain('CRLF response here')

H3 — Doubao test: started_at value is a naive local-time string but normalizeTimestamp behaviour depends on whether a timezone is appended

Line 238:

expect(parsed.started_at).toBe('2026-03-16T10:22:17')

parseDoubaoTranscript sets startedAt = raw directly (no call to normalizeTimestamp) for the Created header value. This is intentional per the source comment ("no timezone info"). However, normalizeTimestamp is called on every message timestamp inside parseIsoTimestamp, which appends +00:00 to bare ISO strings (treating them as UTC). The assertion tests the exact raw value and would catch a regression — that is correct. But the test does not verify that message timestamps are also stored as naive ISO strings (without the Z suffix that normalizeTimestamp would add). Add:

expect(parsed.messages[0]!.timestamp).toBe('2026-03-16T10:22:17')

H4 — Gemini test: title field is never verified after import

The Gemini fixture has no title field in the JSON; the parser falls back to stem (conversation). The manifest and markdown title are never asserted, leaving a gap. The Codex and Claude tests both assert meaningful titles. Add:

expect(manifest.title).toBe('conversation') // stem fallback

And assert cleanContent contains # conversation.

### MEDIUM

M1 — Gemini test: no Gemini JSONL variant tested

The Gemini parser handles both .json and .jsonl extensions (parse.ts lines 295–299). Only the .json path is tested in the integration test. Since the dispatch branches on extname(filePath), the .jsonl code path should have at least one integration test — either here or noted as a gap to be covered in parse.test.ts.

M2 — Doubao test: SQLite title column stores the raw Unicode title, but it is only asserted in the Doubao test, not the Gemini test

Line 271:

expect(rows[0]!.values[0]).toEqual(['doubao', '38416801786598914', 'TypeScript 最佳实践'])

This correctly queries three columns and asserts the full row. The Gemini SQLite query (line 205–207) only checks two columns and uses a string comparison rather than an exact-row assertion:

const rows = db.exec("SELECT client, project FROM transcripts WHERE client='gemini'")
expect(rows[0]!.values[0]).toEqual(['gemini', 'myproject'])

For consistency and completeness, the Gemini SQLite assertion should also check session_id and title.

M3 — Gemini test: no assertion on branch field in parsed or manifest

The Gemini fixture includes branch: 'main'. The test asserts cwd and session_id but never asserts parsed.branch or manifest.branch. This is a coverage gap for the metadata extraction path.

M4 — Both new tests: no verification of the event log (sessions.jsonl)

The Codex test (lines 97–103) verifies that sessions.jsonl exists and contains a parseable event with the correct client. Neither the Gemini nor the Doubao test verifies the event log. The second import in the idempotent test proves that a second import with the same SHA does not append a duplicate — that logic is not exercised for Gemini or Doubao.

M5 — Doubao test: slug from inferProjectSlug is not asserted before being passed to importTranscript

Line 248–249:

const slug = inferProjectSlug(parsed)
const manifest = await importTranscript(parsed, repoRoot, globalRoot, slug, 'none', false)

Since Doubao has no cwd and no client-specific branch in inferProjectSlug, the slug should be 'unknown-project'. This is never explicitly asserted. The Codex test asserts the slug (expect(slug).toBe('myproject')). Adding expect(slug).toBe('unknown-project') makes the intent explicit and prevents silent regressions if the fallback changes.

M6 — Doubao test: manifest fields repo_raw_path, repo_clean_path, repo_manifest_path not verified when mirrorToRepo is false

When mirrorToRepo is false, import.ts sets these three fields to ''. The test never asserts that, meaning a regression where mirroring was accidentally triggered would go unnoticed.

### LOW

L1 — Gemini test: dynamic mkdirSync import instead of static import at the top of the file

Line 138:

const { mkdirSync } = await import('node:fs')

mkdirSync is already available from node:fs in the same test environment. All other tests use static imports at the top of the file. This dynamic import is unnecessary and should be a static import for consistency.

L2 — Gemini test: SQL query uses a string WHERE clause that is client='gemini' but the existing Codex test uses a parameterless query

Lines 205–206 use a hardcoded SQL filter; lines 92–94 use a plain SELECT ... FROM transcripts. Both approaches work, but for consistency and to avoid potential SQL injection patterns in tests that future maintainers might copy, the integration tests should use either prepared statements or consistent patterns.

L3 — Doubao CRLF test: session_id defaults to filename stem (doubao_crlf) rather than a URL-derived ID

The fixture omits a URL: header line, so session_id defaults to the stem doubao_crlf. The test does not assert the session_id value, leaving the fallback behavior untested. Add expect(parsed.session_id).toBe('doubao_crlf') or a similarly explicit check.

L4 — No test for a Doubao file detected via /doubao/ path (vs. doubao_ filename prefix)

inferClient detects Doubao via two conditions: fileName.startsWith('doubao_') or normalized.includes('/doubao/'). Only the filename-prefix path is exercised. The path-directory variant is not tested in either the integration or unit tests.
