2. searchTranscripts rebuilds the FTS index on every call

CREATE_FTS_SQL and REBUILD_FTS_SQL run unconditionally at the start of every searchTranscripts invocation. Because db.js loads the database file from disk each time (the db is not a persistent process-level handle), the IF NOT EXISTS guard on CREATE VIRTUAL TABLE is safe, but the INSERT INTO transcripts_fts(transcripts_fts) VALUES('rebuild') runs every time regardless. For a database with hundreds or thousands of transcripts this is O(N) work per search call, redundant because the data was already in the content table.

The clean fix is to move FTS table creation and initial population into upsertSearchIndex (where writes already happen), and remove the rebuild from searchTranscripts. If you want searchTranscripts to remain standalone (for resilience when the FTS table is missing), the rebuild should be guarded — e.g., only rebuild if the FTS row count is 0. As-is, correctness is fine but the performance overhead grows linearly with transcript count.
