## Code Review: Search Feature

### Summary of Changes

- packages/core/src/transcript/db.ts — Added SearchResult interface, CREATE_FTS_SQL, REBUILD_FTS_SQL, SEARCH_SQL constants, and searchTranscripts() function. FTS5 was changed to FTS4 with content=transcripts.

- packages/cli/src/main.ts — Registered search subcommand.
- README.md — Documented the search command.

---

### CRITICAL

1. FTS4 tokenize=unicode61 is silently unsupported in sql.js (db.ts line 111)

FTS4 only ships with the simple, porter, and icu tokenizers. The unicode61 tokenizer is an FTS5 addition. The sql.js WASM build bundles SQLite without the unicode61 extension, so the CREATE VIRTUAL TABLE call will silently succeed (SQLite falls back to simple) or throw a runtime error depending on the build — neither outcome is documented or tested.

Action: Either remove tokenize=unicode61 to use the simple default, or migrate back to FTS5 if the sql.js build supports it. The switch from FTS5 to FTS4 was presumably made because sql.js's default WASM build omits the FTS5 module; that should be confirmed and documented in a comment. There is no test covering the case where tokenize=unicode61 is rejected.

2. searchTranscripts runs CREATE_FTS and REBUILD_FTS on every search call — potentially catastrophic on a large index (db.ts lines 145-146)

REBUILD_FTS causes a full re-index of the entire transcripts table on each call to searchTranscripts. For a 10,000-row database this blocks the event loop for seconds and bloats WAL. Moving the FTS table creation to upsertSearchIndex (triggered only when data changes) and removing the unconditional REBUILD_FTS from searchTranscripts would be the correct fix. Alternatively, use an FTS with content="" (external content) and rebuild only in the write path.

3. Raw user query passed directly to MATCH without sanitization — query injection (db.ts line 151)

FTS4's MATCH operator uses a mini-query language. Characters like ", , -, (, ) have special meaning. A query such as " (a lone double-quote) will raise a SQLite parse error that propagates as a thrown exception to the CLI, printing a raw SQLite error message to stderr. More problematic: (asterisk-space) or ) can silently corrupt the query semantics. While the binding uses a prepared statement parameter (preventing SQL injection proper), FTS MATCH interprets the parameter value as a query expression — it is not a plain string literal. The error is caught in cmdSearch and printed to stderr, so it is not a security hole, but the user experience is poor and the behavior is undocumented.

Action: Wrap the query in double-quotes to force a phrase search ("${escapedQuery}"), or strip/escape FTS metacharacters before passing them to MATCH. At minimum, document the behavior and add a test for special-character input.

---

### HIGH

4. ManifestEntry has global_manifest_path and cleaning_mode and repo_mirror_enabled fields not present in ALL_COLUMNS (db.ts vs types/src/index.ts)

ManifestEntry declares fields global_manifest_path, cleaning_mode, and repo_mirror_enabled that are not in ALL_COLUMNS (and therefore not in the table schema). The upsertSearchIndex function iterates ALL_COLUMNS to map values, so those three fields are silently dropped on every write. This is not directly a bug in the search feature, but it means the index is not a faithful representation of the manifest. With exactOptionalPropertyTypes enabled, this divergence will not be caught at compile time because the mapping is done at runtime through record[col]. Worth a dedicated fix in the types or a schema migration.

5. rank field is semantically misleading in SearchResult (db.ts lines 104, 123, 162)

The interface declares rank: number but the SQL hardcodes 0 as rank and the comment says "order by relevance is not directly available". A consumer of the JSON output (e.g. an agent integration or external tool) will interpret rank: 0 as meaning all results have equal relevance, which is correct, but the field name implies relevance scoring. This is misleading for a public exported type. Options:
- Remove the field from SearchResult entirely.
- Rename it to rank_score and document 0 as "not computed".
- Replace it with sort_key: string equal to started_at to expose the actual ordering criterion.

The comment in the SQL acknowledges the issue (// FTS4 uses docid (not rank)) but the exported interface makes no mention of it, violating the API contract.

6. searchTranscripts opens the database read-only but runs DDL (CREATE VIRTUAL TABLE, INSERT) against it (db.ts lines 145-146)

The function reads the file with readFileSync and opens an in-memory copy, but running DDL and DML on an in-memory snapshot does not persist. Every call to searchTranscripts therefore creates the FTS table, rebuilds it, queries it, and discards the result — meaning the FTS table is never persisted. This is by design given the sql.js architecture (the DB is in memory), but it means every search pays the full cost of building the FTS index. If the database file is opened by upsertSearchIndex concurrently (e.g., a daemon sync while a search is running), both load the file independently and there is no locking. This is documented nowhere.

### MEDIUM

7. Duplicate import of join and resolve from node:path (cmd-search.ts lines 5-6)

import { join } from 'node:path'
import { resolve } from 'node:path'

These should be a single import: import { join, resolve } from 'node:path'. This is a style/lint issue, but with a strict ESLint config (which the project appears to have) this would likely be flagged.

8. SearchOptions.format is typed as string instead of a union (cmd-search.ts line 16)

format: string

Every other command module (e.g., cmdImport, cmdDiscover, cmdRecall) uses the same loose string type, but the actual accepted values are 'text' | 'json'. An invalid --format foo silently falls through to the text branch. The same issue exists in main.ts where the parseInt(opts.limit, 10) || 20 fallback means --limit 0 is treated as 20. Applying a union type and/or a Commander .choices() call would prevent silent degradation.

9. --root option is documented as unused but still present in SearchOptions and wired through to cmdSearch (main.ts line 146, cmd-search.ts line 13)

The commander option description explicitly says (unused; reserved for future filtering). The root field appears in SearchOptions but cmdSearch never reads options.root. This creates dead code in the public interface and a mismatch between the README (which does not list --root for search) and the actual registered options.

Action: Either remove the option entirely or document it in README. Currently the README omits it, so a user running openmnemo search --help sees an option the docs don't mention.

10. Test for limit uses toBeLessThanOrEqual instead of toEqual (db.test.ts line 285)

expect(results.length).toBeLessThanOrEqual(3)

This passes even if results.length is 0 or 1. Given that 5 records all share the term 'common', the test should assert expect(results.length).toBe(3) to prove the limit is actually enforced at exactly the requested boundary.

11. searchTranscripts test does not assert the rank field value (db.test.ts lines 288-304)

The test for result records have required SearchResult fields checks every field except rank. Since rank is part of the exported SearchResult interface and its constant value of 0 is a design decision, there should be an explicit expect(r.rank).toBe(0) assertion to document and lock in that decision.

12. No test for multi-word queries or FTS MATCH special characters

There are no tests for:
- Multi-word queries ('authentication bug') — FTS4 interprets this as two separate terms by default (AND), not a phrase. A phrase requires "authentication bug". The README example openmnemo search --query "authentication bug" suggests phrase matching, but that is not what happens.
- A query containing ", -, *, or ( — any of these will cause a runtime error or unexpected results.

The shell double-quotes are consumed by the shell and the CLI receives authentication bug as the argument. FTS4 MATCH 'authentication bug' matches documents containing both words independently (AND logic), not the exact phrase. A user who expects "authentication bug" (the phrase) to work differently from authentication bug (two terms) will be surprised. The README should clarify this or the implementation should wrap the query in FTS quote-escaping.
