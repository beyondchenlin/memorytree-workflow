assistant06:13:54

#25 做的功能，通俗讲就是：

本来你执行：

- memorytree init
- 或 memorytree upgrade

它只是在当前仓库里帮你建好 Memory/、AGENTS.md 这些文件。

#25 想再多做一步：
它会顺手把这个仓库登记到你电脑的全局配置里，也就是 ~/.memorytree/config.toml。

这样一来，后台的 heartbeat 系统就会“知道有这个项目了”。

你可以把它理解成：

- init / upgrade：先把房子装修好
- register / quick-start：再去物业登记，接入小区的自动服务

#25 的意思是：你装修房子的时候，我顺手帮你把物业登记也办了。

那我们为什么说要重写，不建议原样合并？

不是因为这个想法没用。
恰恰相反，这个功能方向是有价值的。

问题在于它现在的写法，有点“手伸太长了”。

第一层意思
“用户只想 scaffold 仓库，不一定等于他想把这个仓库加入 heartbeat 管理。”

这句话的大白话就是：

- 用户可能只是想先把项目目录搭起来
- 但他未必已经决定，要让这个项目进入后台自动管理

这里几个英文词我顺手翻一下：

- scaffold：搭脚手架，意思就是“先把基础目录和文件创建出来”
- heartbeat：这里不是医学上的心跳，是“后台定时运行的一套自动任务”
- register：登记、注册
- config：配置文件

所以整句话翻成最通俗的中文就是：

“有些人只是想先把项目框架建出来，不代表他已经同意让后台自动系统接管这个项目。”

为什么这会是个问题？

因为 #25 把两件事混在一起了：

1. 建仓库里的文件
2. 改电脑上的全局配置

而且第二件事不是改当前仓库，是改你整台机器共用的配置。

这就有几个风险：

- 用户没意识到这个命令会改全局设置
- 仓库初始化成功了，但全局配置写失败了，结果变成“半成功半失败”
- 文档现在本来把这两步分开写，用户预期也是分开的
- 测试还没把这个新副作用覆盖好

再用一个更生活化的比喻

现在的 #25 有点像：

“你来买一张地铁卡，我顺手帮你办了整个月票自动续费。”

有些人会觉得方便。
但也有人会说：

“等等，我只是先办卡，还没决定要不要开自动续费啊。”

所以问题不是“自动续费没价值”，而是：

“你不能默认我已经同意这一步。”

所以我们说的“重写”，不是把功能推翻
而是说：

“这个需求可以保留，但实现方式要更克制、更明确。”

比如更合理的改法可以是：

- 方案 1：加一个明确开关
例如 memorytree init --register-project
这样用户自己选

- 方案 2：只在 daemon quick-start 里做注册
因为这个命令本来就是“接入后台自动化”的意思

- 方案 3：自动注册可以做，但失败时不要把整个 init/upgrade 搞成奇怪的半失败状态
最多提示用户：“仓库建好了，但全局注册没完成”

一句话总结
#25 的功能是：

“让 init/upgrade 后，项目自动出现在全局配置里，省掉手动登记。”

我们想重写，不是因为它没用，
而是因为它现在把“建项目框架”和“接入后台自动管理”硬绑在一起了，这对用户来说有点太突然了。
