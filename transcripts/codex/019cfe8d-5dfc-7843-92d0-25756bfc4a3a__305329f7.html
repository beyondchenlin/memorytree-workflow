# AGENTS.md instructions for D:\demo1\papaweb

<INSTRUCTIONS>
# AGENTS.md

## 1) 目标
本文件定义 papaweb 团队开发协作规则，适用于本地团队与所有自动化代理。

## 2) 分支模型
- main：生产/发布分支，只接收经过验证的发布变更。
- dev：日常集成分支，所有功能先合入此分支。
- feature/*：新功能分支，从 dev 切出，合回 dev。
- fix/*：缺陷修复分支，从 dev 切出，合回 dev。
- release/*：发布准备分支，从 dev 切出；验证通过后 PR 合入 main，随后回合 dev。
- hotfix/*：线上紧急修复分支，从 main 切出，先合入 main，再回合 dev。

## 3) 推送与 PR 规则（强约束）
- 禁止直接 push 到 main。
- 禁止直接 push 到 dev。
- 所有日常开发变更必须通过 PR 合入 dev。
- main 只允许来自 release/ 或 hotfix/ 的 PR。
- 每个 PR 至少 1 名 reviewer 审核通过后才能合并。

## 4) CI / E2E 准入规则
- PR 合并到 dev 前，必须满足：
- ci 检查通过（单元测试、静态检查）。
- e2e 检查通过（端到端关键路径）。
- 任一检查失败，禁止合并。
- 禁止以“先合并后补测”为理由绕过。

## 5) TDD（测试驱动开发）规则
- 必须执行 Red -> Green -> Refactor 三步：
- Red：先写测试并确认失败。
- Green：写最小实现让测试通过。
- Refactor：重构代码并保持测试持续通过。
- 修复 bug 时，必须先补回归测试，再改业务代码。
- 新增功能 PR，至少包含：
- 1 个单元测试。
- 1 个集成测试或 E2E 用例（覆盖主流程）。
- 禁止删除/跳过测试来“凑绿灯”。

## 6) PR 必填信息
- 变更目的与范围。
- 验收标准（可观察结果）。
- 测试证据（执行命令与结果）。
