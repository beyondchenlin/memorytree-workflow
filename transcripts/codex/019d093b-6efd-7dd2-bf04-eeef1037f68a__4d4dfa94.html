你这台机器当前状态是：

- ConsoleInput = gb2312
- ConsoleOutput = utf-8

最致命的是最后一项：$OutputEncoding 还是 us-ascii。
在 PowerShell 5.1 里，把 here-string 或管道内容喂给 gh 这种原生命令时，常常会按这个编码发字节，中文直接变 ?。

所以这次 PR body 乱码的真正原因是这种写法：

@'
中文内容
'@ | gh pr create --body-file -

这条链路在 PS 5.1 下很脆弱。

最稳的解决方案
不要把中文内容通过 stdin 管道直接喂给 gh。
统一改成：

1. 先写到 UTF-8 无 BOM 文件
2. 再用 gh ... --body-file <file>

推荐写法：

$utf8NoBom = New-Object System.Text.UTF8Encoding($false)

$text = @'
## 变更目的
这里写中文
'@

$tmp = Join-Path $env:TEMP 'gh-body.md'
[System.IO.File]::WriteAllText($tmp, $text, $utf8NoBom)

gh pr edit 21 --body-file $tmp

这个方案最稳，原因是：
- 不走 PowerShell native pipe 编码
- 不依赖控制台 code page
- 不会出现 BOM 首字符 ﻿

如果你非要走管道
那至少先把当前 session 的编码统一成 UTF-8：

chcp 65001 > $null
$utf8NoBom = New-Object System.Text.UTF8Encoding($false)
[Console]::InputEncoding = $utf8NoBom
[Console]::OutputEncoding = $utf8NoBom
$OutputEncoding = $utf8NoBom

然后再执行 gh。
但即便这样，我还是建议 中文正文一律用 --body-file，不要再用 | gh ...。

长期方案
最好直接改用 PowerShell 7 (pwsh) 做这类自动化。

- Windows 自动化脚本优先用 pwsh

@'中文'@ | gh pr create --body-file -

gh pr create --body "中文很多很多..."

Set-Content -Encoding utf8 body.md ...

最后这一条在 PS 5.1 下通常会写 UTF-8 with BOM，有时会在 GitHub 文本开头出现隐藏字符。
